*   **Rationale**: The foundational components of the `cdqn` ecosystem (Kernel, Persistence, Networking, Scheduler) have been successfully implemented and verified in isolation. The next and final phase of the Minimum Viable Product (MVP) is to unify these components into a single, cohesive `cdqnRuntime` executable that demonstrates the full, end-to-end lifecycle of a KDU.
---

### **1. Guiding Principles for Unification**

*   **Clear Separation of Concerns**: The `main.rs` file will act as the "main orchestrator," responsible for initializing and connecting the services. The services themselves (`Scheduler`, `Persistence`, `Network`) will remain independent and communicate through well-defined interfaces.
*   **Asynchronous by Default**: All I/O operations (Network and Persistence) will be handled in dedicated threads to prevent blocking the main `EntityScheduler` event loop. We will use standard library channels (`std::sync::mpsc`) for safe, thread-to-thread communication.
*   **Security Hardening**: As we unify, we will begin to address the vulnerabilities identified in `check_0002.md`. We will replace `.expect()` and `.unwrap()` calls in our I/O paths with robust error handling and logging.
*   **Configuration Driven**: The runtime's behavior (e.g., whether to run as a server or client, which port to use) will be driven by command-line arguments, making the final executable flexible and testable.

---

### **2. Phase 4: The Unified Sovereign Runtime**

*   **Goal**: To assemble the Kernel, Scheduler, Persistence, and Networking modules into a single, functional `cdqnRuntime` application that can demonstrate the complete lifecycle of a KDU from creation to persistence to network transmission and reception.

#### **Milestone 1: The Runtime Orchestrator & Threaded I/O**

*   **Goal**: To refactor the `main.rs` executable into a true runtime orchestrator that manages all core services in their own threads.
*   **Key Steps**:
    1.  **Create the `Runtime` Struct**: In a new `src/runtime/mod.rs` file, create a main `Runtime` struct that will own the `EntityScheduler`, `Persistence`, and `NodeServer`.
    2.  **Implement Threaded Persistence**: Modify the `C.Persistence` service to run its own dedicated thread. The `EntityScheduler` will communicate with it via a channel (`mpsc::Sender`) to send "write requests" without blocking its own event loop.
    3.  **Implement Threaded Networking**: The `NodeServer` already runs in its own thread. We will create a channel to pass incoming, deserialized KDUs from the network thread back to the main `Runtime` orchestrator.
    4.  **Refactor `main.rs`**: The `main` function will now be extremely simple: it will parse command-line arguments, create the `Runtime` instance, and call a `run()` method on it.

#### **Milestone 2: End-to-End KDU Lifecycle (Local)**

*   **Goal**: To demonstrate the full, local lifecycle of a KDU within a single, unified runtime instance.
*   **Key Steps**:
    1.  **Integrate `KDUFactory`**: The `EntityScheduler` will now use the real `KDUFactory` to create valid, signed KDUs, instead of the dummy ones used in the ping-pong test.
    2.  **Implement "Journaling Middleware"**: The `EntityScheduler`'s `route` function will be modified. Before placing a KDU in a target entity's mailbox, it will first send a clone of that KDU to the `C.Persistence` thread via the channel to be journaled.
    3.  **Update Ping-Pong Test**: The `main.rs` test will be updated. The `Ponger` will now use the `KDUFactory` to create a real "pong" response. We will verify that both the initial "ping" and the "pong" response are written to the journal file.

#### **Milestone 3: End-to-End KDU Lifecycle (Network)**

*   **Goal**: To demonstrate the full, networked lifecycle of a KDU between two instances of the unified `cdqnRuntime`.
*   **Key Steps**:
    1.  **Implement Network Ingress**: The `Runtime` orchestrator will receive KDUs from the `NodeServer`'s channel. It will then use the `EntityScheduler`'s `route` function to deliver the KDU to the correct local entity.
    2.  **Implement Network Egress**: We will introduce the concept of a "remote" FQEI. When the `EntityScheduler`'s `route` function sees a KDU destined for a remote FQEI, it will not put it in a local mailbox. Instead, it will send it to a new `NodeClient` service, which will be responsible for managing outbound connections and sending the KDU over the network.
    3.  **Create the Final Two-Node Test**: The `main.rs` executable will be finalized. It will parse command-line arguments to start in one of two modes:
        *   `--listen`: Starts as a server, running the `Ponger` entity.
        *   `--connect <address>`: Starts as a client, running the `Pinger` entity, and sends an initial "ping" KDU to the server.
    4.  **Verification**: The final test will involve running two instances of the compiled executable from the command line. We will verify that the "ping" is sent from the client to the server, the "pong" is sent back, and that both KDUs are correctly persisted in the respective journal files of both nodes.

*   **Exit Criteria for Phase 4**: A successful demonstration of the two-node test, proving that the unified `cdqnRuntime` can create, sign, schedule, persist, transmit, and receive KDUs in a manner that is consistent, secure, and auditable. This will mark the completion of the sovereign MVP.
