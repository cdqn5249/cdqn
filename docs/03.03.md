## I. Introduction and Continuity from Stage 2 Bytecode

### A. The Security and Capability Validation Gap at the Linguistic Boundary
In the Stage 2 intermediate representation (`QnIR_2` v1.0.4), the compiler co-design successfully verified the **Algebraic Effect Handler System** and the **Wasm Table-Based Capability Shield**, proving that all system-level dispatches can be wrapped inside opaque table-lookup indices (`call_indirect`) to prevent unauthorized, un-auditable background calls [19, 02.08.md].

If a language allows implicit, un-annotated system-calls or un-handled side effects, malicious library authors or speculative execution branches can bypass security boundaries, executing unauthorized background processes (such as sending unrequested telemetry or performing silent software updates) on the physical SoC [15, 30].

## II. Syntactic Formulation of Row-Based Effect Types

To perform rigorous, compositional security analysis over system-level dispatches at the linguistic boundary, we formulate our type system under the algebraic laws of Effect Handlers [13, 26].

### A. Transitive Effect Row Propagation and `@` Annotations
*   **The Derivation Chain:**
    *   *Initial Observation:* To execute system-level I/O or capabilities (like file writing or GPU dispatches), the program must perform side effects [02.08.md].
    *   *Empirical Sources:* Gordon D. Plotkin & John Power (2002) [13] and Gordon D. Plotkin & Matija Pretnar (2009) [26] establish the formal algebraic foundations of monadic side-effects and algebraic handlers.
    *   *Structural Failure:* If a programming language allows implicit, un-annotated system-calls or un-handled side effects, malicious library authors can inject background process calls (such as sending unrequested telemetry), violating our strict **Security ($S$)** and **Modularity ($M$)** invariants [15].
    *   *Required Solution:* Implement **Row-Based Effect Polymorphism** and explicit **`@` (at)** effect rows [15].

Every side-effecting morphism (function) must explicitly declare its required system capabilities in its signature using the `@` operator [03.01.md]:

```qn
-- shalt maps an i64 to the exit state 1 AT the context of wasi_io
shalt : i64_z -> 1 @ wasi_io
```

The compiler's type-checker transitively unifies these effect rows during point-free composition [15, 26]. If a pure morphism $f: A \to B$ composes sequentially (`>>`) with a side-effecting morphism $g: B \to C \;\text{@}\; \text{wasi\_io}$ [03.01.md], the composed morphism $g \circ f$ is statically typed as:

$$g \circ f : A \to C \;\text{@}\; \text{wasi\_io}$$

Any un-declared or un-handled side effect is flagged as a compile-time capability violation, blocking code generation [15].

### B. The `handle-with` Block for Modular Effect Interception
To allow developers to intercept abstract dispatches and route them to target-specific implementations, $QnLang_2$ introduces the structured **`handle ... with ...`** block [26]:
1.  **The Handler Block Syntax:** The abstract operations performed by the inner monoidal pipeline are caught and bound strictly at the boundary of the block [26]:
    ```qn
    handle
      (add2 // id) >> mul2 >> shalt
    with
      wasi_io = standard_wasi_handler
    ```
2.  **The Monoidal Separation:** We propose that the `handle` block acts as a strict, lexically closed scope [26]. The compiler lowers this block by registering the handler's entry point directly into the private, immutable WebAssembly Table (`Table 0`), completely separating the definition of the side-effect from its physical, sandboxed execution [19, 02.08.md].

## III. Information-Flow Control (IFC) and Constant-Time Tagging

To prevent physical registers from exposing private data to microarchitectural timing variations, the language implements static Information-Flow Control (IFC) type-tagging [29, 30].

### A. Static Security Tagging with `#constant_time`
*   **The Derivation Chain:**
    *   *Initial Observation:* Program arithmetic executing on sensitive registers must not leak states via microarchitectural timing variations [02.04.md].
    *   *Empirical Sources:* Iulia Basys, Maximilian Algehed, Alexander Sjösten, & Andrei Sabelfeld (2022) [67] and Liyan Huang, Junzhou He, Chao Wang, & Weihang Wang (2025) [29] establish the formal security lattices of information-flow control in sandboxed WebAssembly environments.
    *   *Structural Failure:* If the compiler cannot distinguish secret data from public metadata, it may compile secret-dependent math into standard conditional branching instructions (`if/else` or conditional jumps), leaking secrets through the CPU's branch predictor and timing side-channels [29].
    *   *Required Solution:* Implement **Information-Flow Control (IFC)** via compile-time **`#constant_time`** compiler attributes [67].

A developer declares a sensitive wire or morphism inside $QnLang_2$ by applying the `#constant_time` prefix attribute [03.01.md]:

```qn
#constant_time
shalt : i64_z -> 1 @ wasi_io
```

The compiler's parser reads this attribute and tags the wire's entry in the Symbol Table with a strict `Secret` security label, while un-annotated wires are tagged as `Public` [15, 67].

### B. Compiling to Branch-Free Wasm `select` Instructions
The compiler's type-checker transitively propagates this `Secret` label down the entire dependency chain of the Symmetric Monoidal Category (SMC) hypergraph [50]. 
1.  **The Information-Flow Guard:** If the type-checker detects that a `Secret` wire is used as an input to a loop boundary, a conditional jump (`if`), or a memory address offset (which would leak state through the cache), **it immediately aborts compilation with an IFC security violation**, completely preventing side-channel leakage at the compiler boundary [15, 50, 67].
2.  **The Branchless Lowering:** Any data-dependent choice operating on a wire carrying the `Secret` label is strictly compiled to WebAssembly's native branch-free **`select`** instruction [2, 29]. Wires carrying the `Public` label (such as the stack pointer `$sp` or the capability handle) are allowed to use standard, structured conditional branch instructions (`if/else`), with the JIT-compiler injecting hardware speculative barriers (`FENCE` / `CSDB`) immediately after the branch to prevent Spectre-style speculative bypass exploits [30, 66].

## IV. Thermodynamic Constraint Annotations and Bounded Costs

To prevent the compiler from generating un-truncated math at deep Zoom scales that could trigger physical hardware-level thermal throttling, $QnLang_2$ implements Thermodynamic Constraint Annotations [33].

### A. Capping Computational Temperature with `#constant_temperature`
We propose to introduce the **`#constant_temperature`** compiler attribute to statically clamp the Zoom scale $z$ [01.5.md, 02.01.md].
*   **The Annotation:** Used to annotate real-time loops and high-frequency transaction blocks [03.01.md].
*   **The Static Check:** The compiler's type-checker verifies that the Zoom scale $z$ and its corresponding remainder thunks ($r_z$) are bounded by a fixed, pre-calculable complexity [03.01.md]. If the compiler calculates that the block’s computational temperature ($T_{\text{comp}} \propto \partial \Phi / \partial z$) could spike dynamically during runtime, **the type-checker blocks compilation at the syntactic boundary with a thermodynamic capability violation error**, preventing hardware-level thermal accumulation at the source of code generation [01.5.md, 02.01.md, 33].

### B. Static CPU-Cycle Resource Enforcement with `#bounded_cost`
We propose to introduce the **`#bounded_cost(C)`** compiler attribute to statically cap the maximum physical CPU cycles ($C$) that the compiled monoidal graph can consume [03.01.md, 58]:
*   **The Cost Model:** The Static Cost Functor ($\mathcal{K}$) in Chapter 2.09 profiles the dataflow, calculating the precise CPU cycle and energy footprint of each monoidal block [02.01.md, 58].
*   **The Decision:** If the total computed cost exceeds the declared bound $C$, the compiler rejects the program at compile-time [02.01.md]. This allows the operating system to allocate exactly the required resources, run the task at native register speed with zero runtime gas checks, and immediately enter its deep sleep state, preserving physical hardware lifespans [1, 33].

## VI. Glossary of Terms for Non-Technical Readers

We define the foundational concepts used in this specification to ensure interdisciplinary readability [1].

### A. Substructural and Effect Row Terminology
*   **Row-Based Effect Polymorphism:** A type-system feature that dynamically tracks and unifies the system capabilities required by a program, ensuring that any un-declared or un-handled side effect is caught and blocked at compile-time.
*   **Algebraic Effect Handler:** A modular code block (`handle ... with ...`) that intercepts abstract system dispatches and routes them to target-specific implementations, separating the definition of a side-effect from its execution.
*   **Information-Flow Control (IFC):** A security mechanism that tracks the flow of sensitive data through a program, statically guaranteeing that secret variables cannot leak or influence public control-flow paths.

### B. Capability and Thermodynamic Terminology
*   **Wasm Table:** A sandboxed WebAssembly memory array that stores function references securely, preventing guest programs from faking system-calls or executing unauthorized background processes.
*   **Computational Temperature ($T_{\text{comp}}$):** The thermodynamic rate of change of our physical hardware resource consumption relative to the Zoom scale, used by the compiler to prevent hardware-level thermal throttling.
*   **Sovereign Code Economy:** A decentralized, peer-to-peer data-transit system where importing packages is modeled as a monoidal functor that negotiates licensing covenants and settles transaction fees natively via $Qn$ ledger primitives under cryptographic, hardware-anchored provenance.
