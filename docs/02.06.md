### Abstract
At the heart of this repository lies a fundamental, interdisciplinary inquiry: **"Can numbers abstract anything?"** 

This paper establishes the sixth specification paper in the CDQN-02 series, designated as `02.06.md` [10]. Building upon the $Wat_{\text{safe}}$ stack-only execution mapping developed in Chapter 2.03 and 2.04 [2], we address the physical and microarchitectural frictions of allocating and managing heap-based memory variables (such as lazy-evaluated remainder thunks and persistent ledgers) on standard hardware platforms [2, 9]. We propose the mathematical and systems-programming specification of the **$p$-adic Tree-Structured Slab Allocator** [10]. 

To bypass the un-typed linear memory security vulnerabilities of WebAssembly [34, 35], we compartmentalize the heap into disjoint, nested **$p$-adic memory balls** [8, 30]. We define a trie-like, prefix-based indexing scheme to resolve memory offsets statically and deterministically, replacing the non-deterministic worst-case latencies of legacy hashmaps and pointer-chasing structures [34, 38]. Furthermore, we utilize linear and affine type constraints ($r_z : Q(S_z) \multimap Q(S_{z+1})$) to execute in-place register-overwrites and deallocations in $O(1)$ constant time [9, 51]. Finally, to prevent guest-level pointer arithmetic and forgery, we wrap our memory stalks strictly in opaque, uninterpreted WebAssembly capability handles (`externref`), mapping them directly to hardware-enforced CHERI capabilities when executing natively under the **cdqnK** kernel [36, 37, 63].

## I. Introduction and the Unmanaged Linear Memory Gap

In Chapter 2.03 (CDQN-02.03, v1.0.2) and 2.04 (CDQN-02.04, v1.0.2), we formulated the **$Wat_{\text{safe}}$ Stack-Only Execution Mapping**, verifying that the core $2$-adic arithmetic operations (`add2`, `mul2`) can execute natively on the CPU stack with zero heap-memory allocation [2]. This stack-only constraint provides complete structural immunity to memory-corruption exploits, serving as the secure foundation for the Stage 0 Seed ($QnIR_0$) [2, 34].

## II. The p-adic Ball Containment Memory Layout

To eliminate the microarchitectural cache-miss storms, non-deterministic search latencies, and pointer-chasing bottlenecks of traditional nested memory structures [3, 30], we model our heap address space strictly as a non-Archimedean ultrametric space [8].

### A. The $p$-adic Memory Ball
Let $p$ be a prime number. Under the mathematical definitions of $p$-adic metric spaces, any two $p$-adic balls in $\mathbb{Q}_p$ are either completely disjoint or one is completely contained within the other [8]. We map this non-overlapping geometric property directly to the physical WebAssembly linear memory heap [2, 8].

We divide the Wasm flat contiguous memory array into a hierarchical sequence of **$p$-adic memory balls**, where each ball represents a strictly bounded memory compartment [8, 13]:

$$B(x, \; p^{-z}) = \{ y \in \mathbb{Q}_p \mid \|y - x\|_p \le p^{-z} \}$$

Where:
*   $x$ is the base center address of the compartment [8].
*   $p^{-z}$ is the maximum allowable physical size (the radius) of the compartment, parameterized by the Zoom scale depth $z \in \mathbb{N}$ [8, 38].

### B. Trie-Like Prefix Address Resolution
Because the $p$-adic address space is totally disconnected and hierarchical, every data stalk has a single, unique coordinate path [8, 47]. 

Finding or resolving the memory offset of a $Qn$ stalk does not require traversing dynamic linked lists or running non-deterministic hash-table lookup probes [38, 40]. The base address of a stalk is resolved **statically and deterministically** by parsing the prefix of its $p$-adic digit valuation [38, 47]:

$$\text{Offset}(X) = \sum_{i=0}^{z} a_i p^i \times S_{\text{slab}}$$

Where:
*   $a_i \in \{0, 1, \dots, p-1\}$ is the $p$-adic digit coordinate [8].
*   $S_{\text{slab}}$ is the fixed, hardware-aligned size of the physical memory slab (such as a 64-byte L1 cache-line or a 4KB page) [33].

```
                     THE p-ADIC PREWRITE ADDRESS PATH
    [ Target Stalk: 3-adic coordinate path 0.12 ]
                           │
                           ▼ (Deterministic digit-path traversal)
    Level 0:  [ Root memory slab: Offset 0 ]
                           │
    Level 1:  └──► [ Sub-Ball 0: Offset 0 ]
                           │
    Level 2:       └──► [ Sub-Ball 0.1: Offset 64 ]
                           │
    Level 3:            └──► [ Stalk Location: Offset 128 ] (O(z) worst-case resolution)
```

This prefix-based path routing behaves as a deterministic trie structure [38]. Unlike legacy hashmaps, there are **zero hash collisions**, and the worst-case lookup time is strictly and mathematically bounded by the finite Zoom scale depth: **$O(z)$ [38, 40]**. This deterministic search bound satisfies the hard real-time and SMR execution constraints of the operating system [7, 30].

## III. The p-adic Tree-Structured Slab Allocator

We specify the resource-management algorithms of the Stage 1 Allocator, demonstrating how linear type invariants enable constant-time deallocation [9].

### A. The Linear Heap-Resource Invariant
Every allocated remainder thunk $r_z$ inside the Wasm linear memory is typed as a **single-owner linear resource** ($\multimap$) [9]:

$$r_z : Q(S_z) \multimap Q(S_{z+1})$$

The compiler ($QnLang_1$) enforces that this memory pointer has exactly one active reference in the symbol table [9].

### B. Constant-Time In-Place Overwrites
When the system deepens its calculation precision ($z \to z+1$), the active thread must update the remainder [10]. 
*   **The Overwrite:** Instead of allocating a new heap memory block to store $r_{z+1}$ and leaving the old $r_z$ block to be collected by a garbage collector, the allocator performs an **in-place overwrite** of the existing memory cells at the exact same physical offset [9, 66].
*   **The Deallocation:** Because the linear type-checker proves that the old state $r_z$ is consumed exactly once by this update, the compiler generates a simple pointer decrement [9]. The old memory block is immediately reclaimed at the register boundary in **$O(1)$ constant time** with zero heap fragmentation and zero pointer-chasing latency [9, 66].

### C. Thread-Local Stalk Isolation
To prevent performance degradation on multi-core SoCs, the compiler enforces **Thread-Local Stalk Isolation** [65].
*   The $p$-adic allocator assigns each CPU core a private, independent memory sub-ball.
*   Mutations of the remainder are executed exclusively within the local L1 cache-line of a single core.
*   Because the memory is completely isolated and never shared across cores during its active, unforced state, the MESI cache-coherence protocol never triggers cross-core invalidation broadcasts [65].
*   This eliminates "cache-coherence storms," preserving the $O(1)$ register execution speed on multi-core substrates [65, 9].

## IV. Software-Enforced CHERI Capability Safeguards

To prevent guest-level software from executing malicious pointer-manipulation or out-of-bounds exploits inside the un-typed WebAssembly linear memory array, the allocator implements software-enforced CHERI capability safety [34, 36].

### A. Spatial Safety via Compartmental Bounds Checking
The allocator compartmentalizes Wasm linear memory by injecting strict bounds-checking instructions at the boundaries of each $p$-adic memory ball [36, 37]:
1.  **The Bounded Allocation:** When a $Qn$ stalk is allocated inside the ball $B(x, p^{-z})$, the allocator seals the allocation bounds into the capability register [36, 37].
2.  **The Compilation Check:** For every memory read or write instruction generated in $QnIR_1$ (`i64.load` or `i64.store`), the compiler statically verifies that the target offset falls strictly within the allocated boundary [36, 37]:
    $$\text{Offset} \ge \text{Base} \quad \text{and} \quad \text{Offset} + \text{Size} \le \text{Base} + \text{Limit}$$
3.  **The Trapping Guard:** If an instruction attempts to read or write data outside these boundaries, the virtual machine immediately throws a fatal **trap**, aborting the guest execution instantly before any spatial memory corruption can occur [2, 37].

### B. Temporal Safety via Capability-Wrapped Handles (`externref`)
To prevent temporal exploits (such as use-after-free or dangling-pointer attacks), **the $QnIR_1$ bytecode completely forbids exposing raw integer pointers to the guest program [2, 36].** 

All heap-allocated memory stalks and resources are wrapped strictly inside opaque, uninterpreted WebAssembly reference handles: **`externref`** [2, 36].

```
                     THE OPAQUE REFERENCE BOUNDARY
    [ Guest QnIR_1 Bytecode ] ───► References data via: [ externref Handle ]
                                                              │
               ┌──────────────────────────────────────────────┴──────────────────────────────────────────────┐
               ▼ Forbidden Guest Operations:                                                   ▼ Enforced Security Bounds:
      - NO Pointer Arithmetic                                                         - Reference is unforgeable [36]
      - NO Address-of (&) operators [37]                                              - Stored securely in Wasm Tables [2]
      - NO Typecasting to raw integers [37]                                           - Traps instantly on misuse [2]
```

Under WebAssembly reference types specifications, an `externref` has **no in-memory representation** inside the guest’s linear memory array [2, 37]. The guest code is physically restricted from [2, 37]:
*   Taking the address of a handle.
*   Casting or converting a handle into a raw integer pointer.
*   **Performing pointer arithmetic** of any kind.

The handles can only be passed by value, stored in private Wasm Tables, or passed to host-defined capability-handlers (such as our `wasi_io` or `cdqnK` kernel handlers) [2, 14, 15].

When executing natively on bare-metal silicon under the `cdqnK` kernel (Option A), these `externref` handles map directly to **hardware-enforced CHERI capabilities** [36, 63, 64]. The CPU's capability unit performs real-time, hardware-level tag and bounds verification. Any attempt to forge an address or reuse an expired capability clears the hardware tag bit, triggering an immediate CPU trap [36].

## V. [Empirical Horizon] and Downstream Roadmapping

*While the $p$-adic tree-structured slab allocator and software-enforced CHERI capability bounds mathematically guarantee spatial and temporal memory safety, we have only defined this as a static, single-threaded allocation model.*

*We have not yet modeled the microarchitectural caching penalties, register-shuffling latencies, and thread-synchronization overhead of executing the parallel $2$-adic hardware thread and the symbolic $5$-adic software cofactor thread. To resolve this, we hypothesize that the system must implement the **Coprime Register Split and 5-adic Emulation Thread** mechanism, which will be the sole subject of **Chapter 2.06a**. Prototyping this parallel splitting within the $QnIR_1$ compilation target is required to verify the caching and performance footprints on legacy hardware (Option C) [33].*

## VI. Glossary of Terms for Non-Technical Readers

*   **Linear Memory:** A contiguous, flat array of raw bytes used by WebAssembly as its internal heap memory. It lacks native type-tagging or safety boundaries at the bytecode level.
*   **$p$-adic Memory Ball:** A mathematically isolated memory compartment inside Wasm linear memory. Its boundaries are determined by $p$-adic metric equations, ensuring allocations are disjoint.
*   **CHERI Capabilities:** A hardware-software security model that replaces raw memory pointers with secure "capabilities" containing base addresses, strict bounds, and unforgeable tag bits to prevent memory-corruption exploits.
*   **`externref` (External Reference):** An opaque WebAssembly reference type representing an external object. It cannot be converted to an integer, and the program cannot execute pointer arithmetic on it, ensuring absolute reference safety.
*   **Thread-Local Stalk Isolation:** A multi-core optimization technique where memory allocations are pinned to single cores, preventing cross-core cache invalidation broadcasts and maximizing processing speed.
