## Abstract

This paper presents an exploratory, speculative design audit and hardening analysis of the proposed **Quang Number (Qn) primitive** [Empirical Horizon]. Building upon the structural abstractions verified in `01.3` (v1.1.0), this paper focuses strictly on evaluating whether the Qn primitive's Capability-Constraint Tuple ($q(z) = \langle \Phi(z), \mathcal{R}(z), \mathcal{K}(z), \mathcal{C} \rangle$) is sufficiently hardened against microarchitectural, algebraic, and cryptographic attack vectors before proceeding to formal mathematical axiomatization in Series 02. 

We limit our audit to the register-packed, software-emulated primitive layer itself, excluding broader system compile-time and operating-system execution environments. 

Through a comparative analysis of real-world low-level and high-level programming languages (C, Rust, Fortran, Java, JavaScript, and CHERI), we investigate whether the Qn primitive can natively abstract secure execution patterns without introducing any new variables or metadata layers. We propose a series of speculative, non-binding operational workflows to harden the primitive: (1) **The Euclidean Division Invariant [Empirical Horizon]**, which isolates sign representation strictly within the quotient, keeping the 32-bit remainder unsigned; (2) **The Scaled Remainder Guarding Workflow [Empirical Horizon]**, designed to prevent arithmetic register overflows during dynamic zoom refinement; (3) **The Object-Capability Register Isolation [Empirical Horizon]** model, which mathematically secures 16-bit compressed tokens against spatial aliasing and brute-force attacks; and (4) **The Constant-Time Predicate Blending [Empirical Horizon]** pipeline, which is conjectured to neutralize data-dependent microarchitectural timing side-channels and ALU early-out logic on commodity consumer CPUs.

## Section I: Consistency Audit of the Compact 128-Bit Primitive Architecture

### 1.1 The Compaction Boundary & Memory Wall Invariants
We review the physical constraint where standard consumer CPUs retrieve data from memory in fixed 64-byte (512-bit) cache lines [1.30, 1.39]. To preserve execution density on standard architectures, we explore the consistency of our 128-bit compacted register layout:

$$
\text{Size}(q(z)) \le 128 \text{ bits}
$$

We evaluate whether packing the quotient, remainder, constraints, and capabilities into a single 128-bit contiguous vector lane can successfully maintain high cache density. Under this model, we hypothesize that the primitive fits natively within standard CPU vector registers (e.g., packing exactly 2 primitives per 256-bit AVX2 register or 4 per 512-bit AVX-512 register), preventing cache-line splits and register spill-loads under high register pressure [1.11, 1.30, 1.39].

This compaction design is conjectured to address the physical limits of the Memory Wall. By ensuring that a single SIMD lane can process the complete mathematical and security metadata of a Capability-Constraint Tuple in parallel, we speculate that we can eliminate the independent memory lookups that degrade performance in legacy pointer-based security architectures [1.1, 1.30, 1.50].

### 1.2 The Unified Native Mapping Hypothesis
We explore the hypothesis that we can represent complex runtime exceptions (such as **Division-by-Zero** and **Arithmetic Bit-Overflow**) natively as boundary breaches within the existing Constraint Set ($\mathcal{K}(z)$) [1.26, 1.50]. 
*   We analyze why this unified mapping is conjectured to eliminate the need for dedicated divisor exception flags, overflow registers, or operating-system context switches [1.2, 1.5].
*   We speculate on how this design maintains a perfect, minimal footprint in active storage, satisfying our core constraints of **Efficiency** and **Security** simultaneously [1.11, 1.30].

To model division-by-zero without introducing dedicated exception flags or signals, we draw from the algebraic framework of **Wheel Theory** ($\mathbb{W}$) [1.59]. In a wheel, division by zero is not an undefined operation; instead, it projects to a defined singular "infinity" ($\infty$) or "null" ($\bot$) element [1.59]. 

We hypothesize that inside our compact 128-bit primitive, the 16-bit Constraint Set $\mathcal{K}(z)$ can designate a reserved, singular "Poison Boundary" bit-pattern (analogous to the NaN payload propagation in IEEE 754-2019 [1.60]):

$$
\text{If } b = 0 \implies B(z) = \emptyset \equiv \text{Poison Boundary Pattern}
$$

If a dynamic division operation yields a zero divisor, the constraint check ($\chi_{\text{con}}$) instantly detects the null/poison boundary pattern, forcing the target register to collapse to the poison state $q_\emptyset$ [1.50, 1.60]. This is conjectured to handle division-by-zero natively as a standard boundary breach within the existing registers, preventing CPU traps [1.39, 1.59].

## Section II: Hardening the Algebraic Plane: Euclidean Division and Overrun Invariants

### 2.1 The Euclidean Division Invariant
We propose the **Euclidean Division Invariant [Empirical Horizon]** to resolve sign-bit allocation ambiguities for negative fractions and prevent representation drift [1.45, 1.46]:

$$
\forall a, b \in \mathbb{Z} \quad (b \neq 0), \quad a \cdot 2^z = \Phi(z) \cdot b + R(z) \quad \text{such that } 0 \le R(z) < |b|
$$

We hypothesize that by isolating the sign bit strictly within the 64-bit signed integer quotient $\Phi(z)$, the dynamic remainder $R(z)$ remains a pure, unsigned 32-bit magnitude ($R(z) \ge 0$). This design is conjectured to prevent sign-bit pollution within the remainder register, allowing standard vector registers to execute exact real arithmetic with zero bit-shifting modifications [1.11, 1.49].

This sign-isolation model is speculated to simplify vector ALU pipeline execution. By ensuring the remainder numerator is always non-negative, we hypothesize that the local platform's translator can map arithmetic operations to standard, unsigned vector instructions, bypassing the complex branching logic required to handle signed remainders in legacy exact real arithmetic libraries [1.39, 1.46].

### 2.2 The Scaled Remainder Guarding Workflow
We formulate a speculative, non-binding workflow to prevent the 32-bit remainder numerator from overflowing when scaling by a large zoom increment $2^{z_2 - z_1}$ [1.39]:

$$
\text{Assert: } z_2 - z_1 \le 32 - \text{clz}(R(z_1))
$$

Where $\text{clz}$ is the hardware-level "count leading zeros" instruction [1.39]. We investigate two tentative execution paths when this invariant is breached: (1) **Iterative Micro-stepping**, which splits the refinement into sequential, bounded micro-steps; and (2) **Temporary Register Elevation**, which elevates the intermediate calculation to a 64-bit scratch register, performing the bitwise shift, and then packing the final refined remainder back into the standard 32-bit slot [1.39, 1.50].

This guarding workflow is designed to prevent silent calculation corruption. By enforcing a strict pre-arithmetic check on the shift size, we speculate that the system can intercept potential remainder overruns before they execute on the ALU, guaranteeing that all progressive refinements remain mathematically consistent and free of silent overflow errors [1.15, 1.44].

## Section III: Hardening the Verification Plane: Object-Capability Table Isolation

### 3.1 The Object-Capability Register Isolation Model
We explore the **Object-Capability Register Isolation [Empirical Horizon]** model, which treats the 16-bit Semantic Namespace Token ($\mathcal{N}$) as a local offset within a private, thread-isolated capability table [1.10, 1.50]. Under this model, the capability is unforgeable because the local CPU's thread-segment registers physically isolate the table from other processes [1.5, 1.51].

We speculate that this local isolation model eliminates the need for expensive, global cryptographic salting or hashing of capability tokens. By confining the 16-bit token's resolution to a private, thread-local memory segment, we hypothesize that the system can achieve fine-grained domain isolation with zero dynamic runtime check overhead, maintaining strict compatibility with standard CPU memory management units (MMUs) [1.10, 1.39].

### 3.2 CHERI-Backed Token Security & Temporal Safety
We use the CHERI Concentrate [1.50, 1.51] and PoisonCap (2026) [1.57] frameworks to mathematically analyze the security of our 16-bit compressed tokens. We speculate on how:
*   Static, local dictionary mappings can resolve these tokens at compile-time ($\Phi_{\text{static}}$), rendering dynamic runtime address searching unnecessary and preventing address collisions [1.15, 1.50].
*   Temporal safety (preventing Use-After-Free or spatial aliasing) can be achieved by utilizing our **temporal validity epoch ($\mathcal{E}$)** as a temporal capability check, ensuring expired tokens immediately evaluate to the poison state $q_\emptyset$ [1.57, 1.58].

This compressed capability model is conjectured to provide robust protection against brute-force guessing attacks. By drawing on CHERI's hardware-enforced bounds compression, we hypothesize that a 16-bit namespace token can safely represent a highly partitioned capability space, ensuring that even if an attacker guesses a valid token bit pattern, they cannot execute a cross-domain reference without holding the explicit local segment permission [1.50, 1.51].

## Section IV: Hardening the Invariant Engine: Constant-Time SIMD Predicate Blending

### 4.1 Neutralizing Microarchitectural Timing Leaks
We explore the **Constant-Time Predicate Blending [Empirical Horizon]** model, designed to prevent data-dependent execution variations (such as ALUs processing the poison state $q_\emptyset$ faster or slower than normal values due to subnormal floating-point handling or early-out logic gates) [1.39, 1.55].

We speculate that by structuring the execution pipeline to bypass conditional branches, we can protect the primitive from timing-side channel attacks. This model is conjectured to be highly valuable for cryptographic and security-critical calculations, as it seeks to ensure that the active computational path does not leak structural metadata (such as boundary breaches or authorization failures) to observing processes [1.55, 1.56].

### 4.2 The Branchless Mask Pipeline
We show the exact planned vector instruction trace to execute the final register write-back using the native, opmask-enabled AVX-512 instruction, completely bypassing software-level conditional jumps [1.48, 1.49, 1.61]:

$$
\text{Assembly Trace (AVX-512): } \text{vpblendmd } \%zmm2 \; \{\%k1\}, \; \%zmm0, \; \%zmm1
$$

At the microarchitectural level:
*   `%zmm0` represents the poison vector $V_{\emptyset}$.
*   `%zmm1` represents the computed algebraic output vector $V_{\text{out}}$.
*   `%k1` represents the hardware opmask register containing the final dual-verification predicate $M_{\text{final}}$ [1.49, 1.61].
*   `%zmm2` is the destination register containing the final secure state $V_{\text{final}}$.

We analyze how this sequence is conjectured to execute with **zero conditional branches** in a fixed, data-independent number of clock cycles (exactly $1$ cycle latency on modern x86 Core architectures) [1.39, 1.49, 1.61, 1.62]. This design aims to ensure that the timing and power signature of the operation remains completely flat, preventing side-channel leakage at the silicon level [1.55, 1.56].

By utilizing standard hardware-level vector blending, we speculate that we can achieve high-performance execution without sacrificing security. We hypothesize that the vector unit's physical multiplexer will execute the write-back in constant-time regardless of whether the lanes contain valid mathematical results or are collapsed to the poison state $V_{\emptyset}$, eliminating data-dependent execution times natively at the processor pipeline level [1.39, 1.49, 1.55].

## Section V: Out-of-Band Attestation & Birth Certificate Verification

### 5.1 Bypassing Locked Bootloader Constraints
We analyze the physical constraints of modern consumer electronics, where proprietary operating systems and locked bootloaders strictly block programmatic access to raw secure enclave keys, JTAG interfaces, and SRAM PUF hardware [1.52, 1.53]. We show why we reject live, runtime PUF queries, as they are physically locked during standard runtime execution [1.23, 1.25].

We suggest that attempting to perform dynamic, on-the-fly hardware queries during primary arithmetic operations is a critical engineering anti-pattern. By avoiding runtime cryptoprocessor calls, we speculate that we can eliminate the massive physical latency penalties and operating-system blocks that would otherwise stall the high-speed execution pipeline [1.11, 1.23].

### 5.2 The Speculative Birth Certificate Verification Workflow
We investigate a tentative, non-binding attestation workflow:
*   **Initialization-Time Challenge:** During the local first-load attestation phase, the translator queries the host's standard secure enclave or TPM via a standard, system-provided cryptographic API to generate a stable signed **Birth Certificate** containing the device-unique signature and mathematical bit-offsets [1.11, 1.23, 1.52, 1.61].
*   **Runtime Verification:** At runtime, the local translator reads the pre-compiled Birth Certificate from the Qn's portable archive on disk, passing it to the host’s standard secure enclave API to verify lineage, completely bypassing locked bootloaders and preventing runtime cryptographic stalls during vector math operations [1.11, 1.23, 1.54].

This out-of-band design aims to balance security and usability. By utilizing standard, manufacturer-provided cryptographic APIs to construct the Birth Certificate, we speculate that we can establish a verified physical lineage on locked consumer devices without requiring custom silicon modifications or running afoul of OS-level security policies [1.23, 1.52].

## Section VI: Glossary for Non-Technical Readers

*   **Euclidean Division:** A division method between two integers where the remainder is mathematically guaranteed to be strictly positive or zero, preventing negative remainders from complicating calculations.
*   **Object-Capability (OCap):** A security model where an application cannot access any resource (like a namespace) unless it is explicitly passed an unforgeable "capability token" representing that specific permission.
*   **Constant-Time Execution:** An execution method where an instruction sequence takes exactly the same amount of time to execute, regardless of whether the input values are zero, positive, negative, or invalid, neutralizing timing side-channel attacks.
