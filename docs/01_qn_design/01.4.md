## Abstract

This paper presents an exploratory, speculative design audit and hardening analysis of the proposed **Quang Number (Qn) primitive** [Empirical Horizon]. Building upon the structural abstractions verified in `01.3` (v1.3.0), this paper focuses strictly on evaluating whether the Qn primitive's Capability-Constraint Tuple ($q(z) = \langle \Phi(z), \mathcal{R}(z), \mathcal{K}(z), \mathcal{C} \rangle$) is sufficiently hardened against microarchitectural, algebraic, and cryptographic attack vectors before proceeding to formal mathematical axiomatization in Series 02. 

We limit our audit to the register-packed, software-emulated primitive layer itself, excluding broader system compile-time and operating-system execution environments. 

Through a comparative analysis of real-world low-level and high-level programming languages (C, Rust, Fortran, Java, JavaScript, and CHERI), we investigate whether the Qn primitive can natively abstract secure execution patterns without introducing any new variables or metadata layers. We propose a series of speculative, non-binding operational workflows to harden the primitive across our established security tiers: (1) **The Euclidean Division Invariant [Empirical Horizon]**, which isolates sign representation strictly within the quotient, keeping the 32-bit remainder unsigned; (2) **The Scaled Remainder Guarding Workflow [Empirical Horizon]**, designed to prevent arithmetic register overflows during dynamic zoom refinement; (3) **The Object-Capability Register Isolation [Empirical Horizon]** model, which mathematically secures 16-bit compressed tokens against spatial aliasing and brute-force attacks; and (4) **The Constant-Time Predicate Blending [Empirical Horizon]** pipeline, which is conjectured to neutralize data-dependent microarchitectural timing side-channels and ALU early-out logic on commodity consumer CPUs.

## Section I: Consistency Audit of the Compact 128-Bit Primitive Architecture

### 1.1 The Compaction Boundary & Memory Wall Invariants
We review the physical constraint where standard consumer CPUs retrieve data from memory in fixed 64-byte (512-bit) cache lines [1.30, 1.39]. To preserve execution density on standard architectures, we explore the consistency of our 128-bit compacted register layout:

$$
\text{Size}(q(z)) \le 128 \text{ bits}
$$

We evaluate whether packing the quotient, remainder, constraints, and capabilities into a single 128-bit contiguous vector lane can successfully maintain high cache density. Under this model, we hypothesize that the primitive fits natively within standard CPU vector registers (e.g., packing exactly 2 primitives per 256-bit AVX2 register or 4 per 512-bit AVX-512 register), preventing cache-line splits and register spill-loads under high register pressure [1.11, 1.30, 1.39].

This compaction design is conjectured to address the physical limits of the Memory Wall [1.1, 1.30]. By ensuring that a single SIMD lane can process the complete mathematical and security metadata of a Capability-Constraint Tuple in parallel, we speculate that we can eliminate the independent memory lookups that degrade performance in legacy pointer-based security architectures [1.1, 1.30, 1.50].

Furthermore, to mitigate the latency penalties during persistent storage operations, we explore the prerequisite of **Asynchronous Lineage Logging [Empirical Horizon]** [1.14]. We propose committing persistent write-backs to an asynchronous, circular log in local persistent RAM (PRAM), allowing a background operating-system thread to flush the data asynchronously and completely eliminating synchronous write-stalls in the primary execution loop [1.14, 1.30].

### 1.2 The Unified Native Mapping Hypothesis
We explore the hypothesis that we can represent complex runtime exceptions (such as **Division-by-Zero** and **Arithmetic Bit-Overflow**) natively as boundary breaches within the existing Constraint Set ($\mathcal{K}(z)$) [1.26, 1.50]. 
*   We analyze why this unified mapping is conjectured to eliminate the need for dedicated divisor exception flags, overflow registers, or operating-system context switches [1.2, 1.5].
*   We speculate on how this design maintains a perfect, minimal footprint in active storage, satisfying our core constraints of **Efficiency** and **Security** simultaneously [1.11, 1.30].

To model division-by-zero without introducing dedicated exception flags or signals, we draw from the algebraic framework of **Wheel Theory** ($\mathbb{W}$) [1.59]. In a wheel, division by zero is not an undefined operation; instead, it projects to a defined singular "infinity" ($\infty$) or "null" ($\bot$) element [1.59]. We hypothesize that inside our compact 128-bit primitive, the 16-bit Constraint Set $\mathcal{K}(z)$ can designate a reserved, singular "Poison Boundary" bit-pattern (analogous to the NaN payload propagation in IEEE 754-2019 [1.60]):

$$
\text{If } b = 0 \implies B(z) = \emptyset \equiv \text{Poison Boundary Pattern}
$$

If a dynamic division operation yields a zero divisor, the constraint check ($\chi_{\text{con}}$) instantly detects the null/poison boundary pattern, forcing the target register to collapse to the poison state $q_\emptyset$ [1.50, 1.60]. This is conjectured to handle division-by-zero natively as a standard boundary breach within the existing registers, preventing CPU traps [1.39, 1.59].

## Section II: Hardening the Algebraic Plane: Euclidean Division and Overrun Invariants

### 2.1 The Euclidean Division Invariant
We propose the **Euclidean Division Invariant [Empirical Horizon]** to resolve sign-bit allocation ambiguities for negative fractions and prevent representation drift [1.45, 1.46]:

$$
\forall a, b \in \mathbb{Z} \quad (b \neq 0), \quad a \cdot 2^z = \Phi(z) \cdot b + R(z) \quad \text{such that } 0 \le R(z) < |b|
$$

We hypothesize that by isolating the sign bit strictly within the 64-bit signed integer quotient $\Phi(z)$, the dynamic remainder $R(z)$ remains a pure, unsigned 32-bit magnitude ($R(z) \ge 0$). This design is conjectured to prevent sign-bit pollution within the remainder register, allowing standard vector registers to execute exact real arithmetic with zero bit-shifting modifications [1.11, 1.49].

This sign-isolation model is speculated to simplify vector ALU pipeline execution. By ensuring the remainder numerator is always non-negative, we hypothesize that the local platform's translator can map arithmetic operations to standard, unsigned vector instructions, bypassing the complex branching logic required to handle signed remainders in legacy exact real arithmetic libraries [1.39, 1.46].

### 2.2 The Scaled Remainder Guarding Workflow
We formulate a speculative, non-binding workflow to prevent the 32-bit remainder numerator from overflowing when scaling by a large zoom increment $2^{z_2 - z_1}$ [1.39]:

$$
\text{Assert: } z_2 - z_1 \le 32 - \text{clz}(R(z_1))
$$

Where $\text{clz}$ is the hardware-level "count leading zeros" instruction [1.39]. We investigate two tentative execution paths when this invariant is breached: (1) **Iterative Micro-stepping**, which splits the refinement into sequential, bounded micro-steps; and (2) **Temporary Register Elevation**, which elevates the intermediate calculation to a 64-bit scratch register, performing the bitwise shift, and then packing the final refined remainder back into the standard 32-bit slot [1.39, 1.50].

This guarding workflow is designed to prevent silent calculation corruption. By enforcing a strict pre-arithmetic check on the shift size, we speculate that the system can intercept potential remainder overruns before they execute on the ALU, guaranteeing that all progressive refinements remain mathematically consistent and free of silent overflow errors [1.15, 1.44].

## Section III: Hardening the Verification Plane: Object-Capability Table Isolation

### 3.1 The Object-Capability Register Isolation Model
We explore the **Object-Capability Register Isolation [Empirical Horizon]** model, which treats the 16-bit Semantic Namespace Token ($\mathcal{N}$) as a local offset within a private, thread-isolated capability table [1.10, 1.50]. We analyze how this workflow is enforced depending on the active security tier of the host platform [1.11]:
*   **Tier 1 (Advanced Hardware-Isolated Security):** The system leverages physical, hardware-isolated CHERI tag memory and hardware capability segment registers to physically prevent user-space capability forgery of the 16-bit namespace token $\mathcal{N}$ and 4-bit `PBR_id` handle [1.50, 1.51].
*   **Tier 2 (Best-Effort Software-Defined Security):** The system relies on compile-time SFI (Software-Fault Isolation) segment-boundary validation and static type-relation analyses to prove boundaries statically at build-time, completely preventing out-of-segment table manipulation [1.10].

We speculate that this tiered local isolation model eliminates the need for expensive, global cryptographic salting or hashing of capability tokens. By confining the 16-bit token's resolution to a private, thread-local memory segment, we hypothesize that the system can achieve fine-grained domain isolation with zero dynamic runtime check overhead, maintaining strict compatibility with standard CPU memory management units (MMUs) [1.10, 1.39].

To bypass the memory copying and context-switching overhead of managing these tables, we propose the **Segmented Lineage Base Registers (LBR) [Empirical Horizon]** prerequisite [1.5, 1.39]. On an OS context switch, the scheduler *only* swaps a 64-bit base register pointer [1.5], achieving constant-time ($O(1)$) context switching and preventing memory bus saturation.

### 3.2 CHERI-Backed Token Security & Temporal Safety
We use the CHERI Concentrate [1.50, 1.51] and PoisonCap (2026) [1.57] frameworks to mathematically analyze the security of our 16-bit compressed tokens. We speculate on how temporal safety (preventing Use-After-Free or spatial aliasing) can be achieved by utilizing our **temporal validity epoch ($\mathcal{E}$)** as a temporal capability check [1.57, 1.58]:
*   **Tier 1 (Advanced Hardware-Isolated Security):** The system leverages hardware-enforced epoch-revocation checks and hardware-isolated capability table sweeps to guarantee that expired tokens immediately evaluate to the poison state $q_\emptyset$ [1.57].
*   **Tier 2 (Best-Effort Software-Defined Security):** The system relies on compiler-inserted software boundary checks and lifetime assertions at volatile interfaces, ensuring that stale memory-mapped registers cannot be accessed after epoch expiry [1.10, 1.58].

This compressed capability model is conjectured to provide robust protection against brute-force guessing attacks. By drawing on CHERI's hardware-enforced bounds compression, we hypothesize that a 16-bit namespace token can safely represent a highly partitioned capability space, ensuring that even if an attacker guesses a valid token bit pattern, they cannot execute a cross-domain reference without holding the explicit local segment permission [1.50, 1.51].

## Section IV: Hardening the Invariant Engine: Constant-Time SIMD Predicate Blending

### 4.1 Neutralizing Microarchitectural Timing Leaks
We explore the **Constant-Time Predicate Blending [Empirical Horizon]** model, designed to prevent data-dependent execution variations (such as ALUs processing the poison state $q_\emptyset$ faster or slower than normal values due to subnormal floating-point handling or early-out logic gates) [1.39, 1.55].

To prevent the transitive propagation of corrupted or flipped register states without introducing heavy runtime hashing, we propose the **Algebraic Parity Invariant [Empirical Horizon]** [1.26]. We hypothesize that the register's 16-bit Constraint Set $\mathcal{K}(z)$ must include a **3-bit algebraic parity or Error-Correcting Code (ECC)**. The ALU calculates the 3-bit parity of the output quotient $\Phi_{\text{out}}(z)$ and remainder $R_{\text{out}}(z)$ in parallel [1.26, 1.39]. If a physical bit-flip occurs (e.g., from Rowhammer [1.30]), the computed parity conflicts with the register's embedded parity, causing the constraint check ($\chi_{\text{con}}$) to instantly evaluate to $0$, collapsing the register to $q_\emptyset$ [1.11, 1.15, 1.26].

We speculate that by structuring the execution pipeline to bypass conditional branches, we can protect the primitive from timing-side channel attacks. This model is conjectured to be highly valuable for cryptographic and security-critical calculations, as it seeks to ensure that the active computational path does not leak structural metadata (such as boundary breaches or authorization failures) to observing processes [1.55, 1.56]. 

Importantly, we propose that this Constant-Time Predicate Blending workflow is **fully shared across both Tier 1 and Tier 2 security models** [1.11]. Because both tiers compile their execution blocks down to standard hardware instructions, they rely on the same native vector registers to achieve branchless, flat-latency execution [1.49, 1.61].

### 4.2 The Branchless Mask Pipeline
We show the exact planned vector instruction trace to execute the final register write-back using the native, opmask-enabled AVX-512 instruction, completely bypassing software-level conditional jumps [1.48, 1.49, 1.61]:

$$
\text{Assembly Trace (AVX-512): } \text{vpblendmd } \%zmm2 \; \{\%k1\}, \; \%zmm0, \; \%zmm1
$$

At the microarchitectural level:
*   `%zmm0` represents the poison vector $V_{\emptyset}$.
*   `%zmm1` represents the computed algebraic output vector $V_{\text{out}}$.
*   `%k1` represents the hardware opmask register containing the final dual-verification predicate $M_{\text{final}}$ [1.49, 1.61].
*   `%zmm2` is the destination register containing the final secure state $V_{\text{final}}$.

We analyze how this sequence is conjectured to execute with **zero conditional branches** in a fixed, data-independent number of clock cycles (exactly $1$ cycle latency on modern x86 Core architectures) [1.39, 1.49, 1.61, 1.62]. This design aims to ensure that the timing and power signature of the operation remains completely flat, preventing side-channel leakage at the silicon level [1.55, 1.56].

By utilizing standard hardware-level vector blending, we speculate that we can achieve high-performance execution without sacrificing security. We hypothesize that the vector unit's physical multiplexer will execute the write-back in constant-time regardless of whether the lanes contain valid results or are collapsed to the poison state $V_{\emptyset}$, eliminating data-dependent execution times natively at the processor pipeline level [1.39, 1.49, 1.55].

## Section V: Out-of-Band Attestation & Birth Certificate Verification

### 5.1 Bypassing Locked Bootloader Constraints
On standard consumer smartphones and laptops, proprietary operating systems and locked bootloaders strictly block programmatic access to raw secure enclave keys, JTAG interfaces, and SRAM PUF hardware [1.52, 1.53]. We propose that direct, live runtime PUF queries are physically locked and structurally inaccessible [1.23, 1.25]. We investigate an alternative model where any physical validation or cryptographic key extraction is handled strictly out-of-band during system initialization [1.23].

This initialization-time restriction is designed to prevent runtime performance degradation. By avoiding dynamic, on-the-fly hardware queries during primary arithmetic operations, we speculate that we can eliminate the physical latency penalties and OS-level blocks that would otherwise stall the high-speed execution pipeline [1.11, 1.23].

We suggest that attempting to perform dynamic, on-the-fly hardware queries during primary arithmetic operations is a critical engineering anti-pattern. By avoiding runtime cryptoprocessor calls, we speculate that we can eliminate the massive physical latency penalties and operating-system blocks that would otherwise stall the high-speed execution pipeline [1.11, 1.23].

### 5.2 The Speculative Birth Certificate Verification Workflow
We investigate a tentative, non-binding attestation workflow, showing how the local platform translator verifies the physical lineage of the primitive while completely preventing runtime cryptographic stalls during vector math operations [1.11, 1.23]:
*   **Static Load-Time Attestation ($\Phi_{\text{static}}$):** When a Qn primitive $q(z)$ is loaded into memory-mapped storage from disk/NVMe [1.14], the local platform translator verifies its **Birth Certificate (BC) [Empirical Horizon]** out-of-band [1.11, 1.23].
    *   *Tier 1 (Advanced Hardware-Isolated Security):* The Birth Certificate [Empirical Horizon] is the signed asymmetric cryptographic package ($\text{BC} = \text{Sign}_{K_{\text{master}}}(\text{PBR})$) generated by the host's Secure Enclave using its master key ($K_{\text{master}}$) [1.23, 1.52, 1.54]. To protect the attestation from the threat of quantum Shor's algorithm, we speculate that the Birth Certificate must be signed using native, **lattice-based ML-DSA digital signatures** [1.78], entirely eliminating classical RSA/ECDSA key exposure.
    *   *Tier 2 (Best-Effort Software-Defined Security):* The translator bypasses the asymmetric check, relying on **SFI compiler verification** to prove boundaries and capability tables statically at build-time, completely preventing out-of-segment table manipulation [1.10]. High-entropy emulated PBR payloads are generated out-of-band using standard CPU random instructions (`RDSEED`) [1.49, 1.61] and clock counters (`RDTSC`) [1.39, 1.61, 1.77].
*   **Lineage Table Isolation:** To prevent user-space forgery of the 4-bit `PBR_id` handles, we hypothesize that the thread-local **Lineage Table ($\mathbf{R}_{\text{lineage}}$)** must be hardware-isolated [1.50, 1.51]. By leveraging hardware capability segment registers or CHERI-style tag memory [1.50, 1.51], any attempt by an untrusted user-space process to modify the 4-bit handles or mutate the table contents directly is physically blocked by the hardware, guaranteeing unforgeable transitive trust.
*   **Lightweight Lineage Hashing:** To prevent execution stalls, the expensive asymmetric signature validation is restricted to this out-of-band load phase [1.39, 1.55]. At runtime, the local translator only verifies the 4-bit `PBR_id` and the causal lineage hash chain ($\mathcal{H}_{\text{lineage}}$) [1.50] of the **Physical Birth Register (PBR) [Empirical Horizon]** using a hardware-friendly, low-overhead hash function (such as **SHA3-256** or **SHAKE256**), executing in a few clock cycles [1.55, 1.74, 1.80].
*   **Transitive Trust during SIMD Execution ($\Phi_{\text{dynamic}}$):** Once loaded into thread-local, hardware-isolated registers [1.50], the output of any vector operation is transitively trusted. The intermediate steps are policed strictly via the branchless, constant-time `vpblendmd` mask multiplexer [1.49, 1.61].

## Section VI: Glossary for Non-Technical Readers

*   **Euclidean Division:** A division method between two integers where the remainder is mathematically guaranteed to be strictly positive or zero, preventing negative remainders from complicating calculations.
*   **Object-Capability (OCap):** A security model where an application cannot access any resource (like a namespace) unless it is explicitly passed an unforgeable "capability token" representing that specific permission.
*   **Constant-Time Execution:** An execution method where an instruction sequence takes exactly the same amount of time to execute, regardless of whether the input values are zero, positive, negative, or invalid, neutralizing timing side-channel attacks.
