## Section I: Abstract & Introduction: Deriving from `01.1`

### 1.1 The Philosophical Handoff from `01.1`
In our foundational green paper, `cdqn-01.1` (v1.1.1), we questioned the semantic-blind nature of classical numbers and hypothesized that a "luminous" numerical system—signified by the etymological root of the Vietnamese word **Quang (光)** representing light and clarity—could act as a direct carrier of abstraction to resolve the physical Memory Wall and the logical Semantic Wall [1.1, 1.2]. By doing so, we explore whether we can eliminate all un-monitored shadows, hidden processes, and un-verifiable memory spaces from the computing environment, leaving no room for doubt or anonymous execution [1.2].

However, as we transition from high-level philosophy to the practical boundaries of computer systems, we must immediately raise a self-critical question: *Are we merely replacing old, fragile software abstractions with equally complex, un-scalable metadata abstractions packed into a new physical register?* If we overload our numerical primitives with too much absolute context, we risk creating a bloated structure that worsens the very memory bottlenecks we seek to escape [1.1, 1.2].

Thus, this paper is designed to systematically explore the physical and logical prerequisites of such a clean-slate design on existing consumer electronics. We do not declare a finalized microarchitecture; rather, we investigate the speculative constraints of the physical media to prove that a hardware-agnostic, constructivist algebraic stack is theoretically feasible on commodity hardware.

### 1.2 The Core Question of `01.2`
To explore the question, *"how can we abstract anything from a numbers system?"*, we must first investigate the foundational prerequisites that must be checked *before starting any design*. We introduce the target layers of the proposed stack—the new coding language (**QnLang [Empirical Horizon]**) and the new operating system (**cdqnOS [Empirical Horizon]**)—strictly as forward-looking conceptual components of our research.

We speculate on whether these upper-layer software components can be designed to natively inherit the safety properties of the underlying algebraic primitive [1.10]. Instead of relying on a massive operating-system kernel to act as a defensive software police engine, we investigate whether cdqnOS [Empirical Horizon] can collapse into a minimalist, hardware-aligned translator of mathematical relations, while QnLang [Empirical Horizon] achieves an exceptionally narrow semantic gap with the hardware [1.2, 1.11].

Our exploration in `01.2` is strictly non-binding and exploratory. We do not assume our target stack is finalized; rather, we treat each layer as an open hypothesis to be rigorously audited against the physical realities of modern hardware. This ensures that our future specifications remain grounded in verified microarchitectural constraints.

### 1.3 The Hardware-Agnostic and Practical Deployment Constraints
We establish the absolute, mandatory prerequisite that the proposed Quang Number (Qn) [Empirical Horizon] primitives must be designed to be as hardware-agnostic as possible, strictly managing our data within standard hardware I/O and memory execution workflows of existing commodity consumer devices [1.14]. This approach aims to bypass the massive financial and technical barriers of custom silicon fabrication, allowing the CDQN stack to be emulated and verified natively on standard, modern processors [1.11].

However, we must confront a major practical deployment constraint: because modern consumer smartphones feature heavily locked bootloaders and proprietary operating system controls, installing a completely new operating system like cdqnOS [Empirical Horizon] on standard mobile devices is functionally impossible. Therefore, the primary practical platforms for running and testing our Proof of Concept (PoC) are limited to **PCs, laptops, and consumer-accessible servers**.

To prevent locking our design to only these highly advanced devices, we explore a **Tiered Security Model [Empirical Horizon]** that scales dynamically based on the host hardware's physical capabilities without requiring custom silicon modifications [1.10, 1.11]. This tiered architecture is conjectured to allow graceful degradation on legacy systems while maximizing hardware-enforced protection on advanced platforms.

### 1.4 Grounding the Physical Latency Gap
To ensure our baseline is grounded in verified systems engineering reality, we analyze the physical performance profile of contemporary memory interfaces. Standard CPU registries calculate data in fractions of a nanosecond, whereas retrieving a single block of data from external main memory (DRAM) requires up to $100$ nanoseconds, a performance bottleneck formally known as the Memory Wall [1.1, 1.30].

This physical latency gap forces modern processors to spend up to $90\%$ of their active clock cycles stalled, waiting for data to travel across the memory bus [1.30]. While hardware manufacturers attempt to mitigate this through massive multi-level cache hierarchies, these physical solutions are reaching hard thermal and power-grid limits [1.3].

Therefore, we hypothesize that our data representations must be designed to maximize cache-line utility entirely in software [1.2, 1.30]. By ensuring that the primitive and its metadata are tightly compacted to fit within a single standard 64-byte vector lane, we speculate that we can eliminate dynamic cache splits and split-loads, extracting optimal performance from existing consumer SoCs.

## Section II: Constraints Imposed by Hardware Physics & Computer Science

### 2.1 The Physical Constraints (The Memory & Power Walls)
When data is executed on standard consumer devices, its throughput is strictly governed by physical hardware limits. Data must travel along physical motherboard copper traces, which are fundamentally constrained by electrical resistance, capacitance, and the speed of light, introducing unavoidable propagation latencies [1.1].

Furthermore, CPUs fetch memory in fixed, aligned blocks (typically 64 or 128 bytes) called cache lines [1.30]. If our data layout is poorly aligned or arbitrarily sized, a single logical calculation will straddle the boundary of two physical cache lines (a cache-line split), forcing the memory controller to execute two slow DRAM fetches instead of one, doubling the latency penalty [1.2, 1.30].

To mitigate these latency penalties during persistent storage operations, we introduce **Asynchronous Lineage Logging [Empirical Horizon]** as a core prerequisite. When writing data to memory-mapped storage (such as NVMe flash), standard synchronous flushes introduce severe write-queue congestion and block the pipeline [1.14, 1.30]. 

We speculate that by committing persistent write-backs to an asynchronous, circular log in local persistent RAM (PRAM), a background operating-system thread can flush the data asynchronously, completely eliminating synchronous write-stalls in the primary execution loop [1.14, 1.30].

### 2.2 The Logical Constraints (The Software Execution Workflow)
In standard software design, developers write code using high-level, semantic-rich abstractions, but the runtime execution on legacy processors is a heavily constrained, slow pipeline of serializations and deserializations [1.2, 1.12]. Because traditional hardware is semantic-blind, software must continuously convert data between human-readable text formats and binary register representations [1.12].

This format-conversion tax routinely consumes up to $30\%$ to $50\%$ of active CPU cycles in standard business application workflows, completely wasting ALU compute pipelines [1.12, 1.38]. To protect memory boundaries, compilers must inject explicit branch instructions before every memory write to verify that the index is within bounds [1.2, 1.39]. This pollutes the CPU's branch predictor and causes frequent, expensive pipeline flushes (15-20 cycles each) [1.2, 1.39].

To bypass the severe memory copying and context-switching overhead of managing thread-local, hardware-protected capability tables [1.57], we propose **Segmented Lineage Base Registers (LBR) [Empirical Horizon]** as a logical prerequisite. Instead of copying large lineage tables on every thread context switch (which takes dozens of nanoseconds [1.5]), we hypothesize that the CPU architecture can expose a dedicated control register [1.39]. 

On a context switch, the operating system *only* swaps a 64-bit base register pointer [1.5], achieving constant-time ($O(1)$) context switching and preventing memory bus saturation.

## Section III: The Agnostic I/O Management Model (What We Can and Cannot Do)

### 3.1 What the Proposed Stack Can Do
We explore how the proposed stack can manage the data layout of Qn primitives [Empirical Horizon] strictly within the I/O pathways of existing hardware [1.14]. For hardware architectures that feature advanced vector extensions (such as standard single-instruction, multiple-data (SIMD) registers), we can explore packaging our data into standard, contiguous vector structures [1.15].

This vector-packing approach is conjectured to allow the local translator to compile scale alignments and boundary checks into branchless, hardware-level mask instructions across the vector registers, preserving exact, constructivist arithmetic on standard silicon [1.11, 1.15]. 

To prevent dynamic execution loops (such as the Collatz trajectory) from violating the static scheduling invariants of Synchronous Dataflow (SDF) compilers [1.15], we propose the **Step-Gated Trajectory Decomposition (SGTD) [Empirical Horizon]** prerequisite:
*   **Static Step Bounding:** We speculate that any register-level operation must be strictly gated to a compile-time static iteration depth (e.g., $k \in \mathbb{N}$ steps) [1.11, 1.15].
*   **Non-Looping Pipeline:** The vector ALU executes exactly $k$ flat, un-rolled steps (shifts and additions) in a branchless pipeline [1.39, 1.49]. This is conjectured to guarantee a fixed-latency, predictable execution cycle that compiles natively into a static, non-blocking SDF node, preventing scheduling stalls.

### 3.2 What the Proposed Stack Cannot Do
We must establish absolute, non-negotiable boundaries of non-feasibility. Our software stack cannot add custom mathematical instructions to physical consumer device ALUs [1.11], nor can it modify or override the hardware-level Page Table structures of the host processor's Memory Management Unit (MMU) [1.5, 1.10].

Furthermore, our stack cannot perform raw, uninitialized Physical Unclonable Function (PUF) queries dynamically on-the-fly during runtime, as active memory cells are populated with live system data and cannot be reset without a full reboot [1.23, 1.25]. 

This physical limitation dictates that any physical validation or cryptographic key extraction must be handled strictly out-of-band during system initialization, completely preventing live, on-the-fly hardware register queries from stalling the primary execution pipeline [1.11, 1.23].

### 3.3 The Core Prerequisite of QnIR
To bridge the gap between high-level language design and low-level machine execution, we introduce the intermediate representation, **QnIR [Empirical Horizon]**, as a mandatory prerequisite of our design [1.15]. QnIR [Empirical Horizon] acts as the stable, hardware-agnostic intermediate translation language [1.15].

Instead of compiling directly to a specific CPU's assembly, QnIR [Empirical Horizon] is designed to preserve the high-level mathematical and semantic *intent* (such as mathematical relations, self-verifying boundaries, and capability tokens) in a structured, un-flattened bytecode [1.15].

When loaded onto a specific target device, the local platform's translator reads this abstract QnIR [Empirical Horizon] bytecode, verifies its invariants out-of-band, and compiles/maps it 1-to-1 into the physical instructions of that specific SoC, achieving native, zero-copy execution with no hardware modifications [1.11, 1.15].

## Section IV: The Tiered Security Model & The Physical Birth Register [Empirical Horizon]

To avoid locking our design strictly to a single class of high-end hardware, we explore a tiered security architecture that scales dynamically based on the host system's physical capabilities.

```
                      Physical Birth Register (PBR) Generation
                      
   Unstable Noise Oscillator ───> [ Edge Detection ] ───> x Noise Events (Duration T_opt)
                                                               │
   Stable System Clock       ───> [ Cycle Counter ]  ───> Count cycles c_i between events
                                                               │
                                                               v
                                                 Entropy Vector: c = [c_1, c_2, ..., c_x]
                                                 Lineage Hash:   H_lineage = Q(n-1) ⊕ c
```

### 4.1 The Physical Birth Register (PBR) Model [Empirical Horizon]
We hypothesize that a secure numerical primitive cannot rely on a static, administrative certificate. Instead, we propose the **Physical Birth Register (PBR) [Empirical Horizon]**, where a Qn is literally "born" from physical hardware entropy. Let the hardware platform contain an unstable noise source (such as thermal noise or oscillator jitter) and a high-frequency system reference clock [1.23, 1.76]:
*   **Entropy Harvesting:** Over an optimized duration $T_{\text{opt}}$, the hardware edge-detection circuitry captures $x$ successive noise event transitions [1.73, 1.76].
*   **Cycle Counting:** The hardware counts the number of system clock cycles ($c_i \in \mathbb{Z}^+$) elapsed between these events, producing an unpredictable, high-entropy seed vector $\mathbf{c} = [c_1, \dots, c_x]$ [1.77].
*   **Causal Chaining:** The primitive is cryptographically chained to its parent state ($Q_{n-1}$) [1.10]:
    $$
    \mathcal{H}_{\text{lineage}} = \text{Hash}(Q_{n-1}) \oplus \text{Hash}(\mathbf{c})
    $$
    To guarantee absolute, post-quantum resistance against Grover's search algorithm, we mandate that this PBR lineage hash ($\mathcal{H}_{\text{lineage}}$) must utilize a minimum **256-bit keyspace** (such as standard SHA3-256 or SHAKE256 [1.80]) rather than symmetric PRF algorithms like SipHash [1.74, 1.75].
*   **The Compaction Interface:** To prevent register and cache bloat, the raw vector $\mathbf{c}$ and lineage hash $\mathcal{H}_{\text{lineage}}$ are stored out-of-band in the thread-local Lineage Table ($\mathbf{R}_{\text{lineage}}$) [1.50], mapped to the register-packed capability set via a compact 4-bit handle index (`PBR_id`) [1.50].

### 4.2 Tier 1: Advanced Hardware-Isolated Security (ASIC/FPGA) [Empirical Horizon]
For advanced computer systems equipped with specialized hardware, the PBR is implemented directly in silicon logic gates:
*   **Physical Noise Tracking:** The hardware features a dedicated physical True Random Number Generator (TRNG) coprocessor that continuously monitors silicon thermal noise and oscillator jitter [1.23, 1.73, 1.76].
*   **Lattice-Based PQC-Attested Secure Enclaves:** During local first-load attestation, the secure enclave uses its boot-time derived master key ($K_{\text{master}}$) to sign the PBR payload, generating the stable **Birth Certificate [Empirical Horizon]** [1.23, 1.52, 1.53]. 
    
    To protect the attestation from the threat of quantum Shor's algorithm, we speculate that the Birth Certificate must be signed using native, **lattice-based ML-DSA digital signatures** [1.78], entirely eliminating classical RSA/ECDSA key exposure.

### 4.3 Tier 2: Best-Effort Software-Defined Security (Commodity Systems) [Empirical Horizon]
For standard commodity consumer devices (such as laptops and smartphones) lacking raw hardware-level noise oscillators or custom silicon, the system degrades gracefully:
*   **Emulated Entropy Birth:** The local platform translator emulates the cycle-counting. It queries standard CPU hardware entropy instructions (`RDSEED`) [1.49, 1.61] and hashes the output with the CPU's high-resolution performance clock counter (`RDTSC`) [1.39, 1.61, 1.77]:
    $$
    \mathbf{c}_{\text{emulated}} = \text{Hash}\left( \text{RDSEED} \oplus \text{RDTSC} \right)
    $$
*   **The Security Boundary:** While this configuration provides robust logical safety against standard programming errors and memory-safety exploits [1.10], we explicitly exclude protection against unpredictable hardware-level zero-day security breaches or physical side-channel exploits [1.11]. This allows the CDQN [Empirical Horizon] stack to remain universally accessible on low-cost consumer hardware, while reserving maximum hardware-enforced protection for Tier 1 environments.

### 4.4 Speculative Post-Quantum Cryptographic Agility (QnPQC) [Empirical Horizon]
To preserve cryptosystem sovereignty without relying on proprietary, closed-source vendor security modules, we propose the prerequisite of **Quang Number Post-Quantum Cryptography (QnPQC) [Empirical Horizon]**. Standard asymmetric schemes (such as RSA or ECDSA) are fundamentally vulnerable to Shor's algorithm, exposing the system to retrospective attestation forgery ("Harvest Now, Decrypt Later") [1.74, 1.75]. 

We speculate that we can construct a completely sovereign, quantum-resistant digital signature scheme natively on top of the Qn primitives [1.10]. However, to preserve the 128-bit compaction boundary and prevent performance-destroying register bloat, we strictly enforce the **Multi-Layer Dimensionality Separation** principle:
*   The Qn primitive itself remains a strict **1D scalar** ($q \in \mathbb{Q}_n$) [Empirical Horizon].
*   High-dimensional lattice modules (polynomial rings over modular rings, e.g., $R_q = \mathbb{Z}_q[x] / (x^n + 1)$ [1.79]) are defined strictly at the **upper software layers (QnLang / cdqnOS)** as multidimensional vectors of 1D Qn primitives [1.15].
*   The local platform translator compiles these high-dimensional lattices into parallel, non-blocking SIMD vector lanes, executing PQC signatures at the physical limit of the silicon.

## Section V: The Genesis Bootstrapping Strategy for QnLang [Empirical Horizon]

### 5.1 The Formal Verification of the Stage 0 Seed [Empirical Horizon]
To ensure that the compilation toolchain of the new coding language (**QnLang [Empirical Horizon]**) is built securely, modularly, and natively without carrying the security vulnerabilities of legacy languages, we propose the **Genesis Bootstrapping [Empirical Horizon]** strategy [1.11, 1.4]. To defeat the "Trusting Trust" backdoor paradox—where legacy compilers silently compile hidden vulnerabilities into newly developed languages—we reject the standard practice of using an existing high-level language (like C, Rust, or Zig) for our initial compiler.

In modern software design, developers leverage advanced AI coding frameworks to generate the bulk of the initial code, shifting the developer's primary workload to rigorous code reviews and verification [1.4, 1.11]. However, manual code reviews are vital but human inspectors are notoriously poor at detecting subtle, compiled-in vulnerabilities or complex logical flaws in generated machine code [1.27, 1.42]. Relying solely on manual human auditing inside the Trusted Computing Base (TCB) introduces severe, un-scalable security risks [1.42].

To guarantee absolute security, we propose integrating **automated formal verification and translation validation** [1.42, 1.43]. Using formal theorem provers (similar to the techniques used in verified compilation systems) [1.42, 1.43], we mathematically prove that the AI-generated binary seed's machine-code bytes are semantically and logically isomorphic to the original, un-compiled specification [1.42]. This mathematically guarantees the absolute security of the Stage 0 seed, completely eliminating any AI hallucinations or hidden compiler-introduced security bugs (CISBs) before execution [1.27, 1.42].

### 5.2 The Three-Stage Compilation Scale [Empirical Horizon]
We analyze how the compiler dynamically scales from low-level systems programming to high-level semantic intent:
*   **Stage 0 (The Binary Seed):** An AI-generated, formally-verified micro-assembler to compile a minimal, safe subset of QnLang [Empirical Horizon] into QnIR [Empirical Horizon] [1.11, 1.15, 1.42].
*   **Stage 1 (Self-Hosted Systems Compiler):** Written in the safe subset of QnLang [Empirical Horizon] and compiled by Stage 0, it natively inherits the system's security invariants and compiles the core execution platform [1.10, 1.11].
*   **Stage 2 (The Semantic/Agentic Compiler):** Evolved to compile high-level, abstract semantic loops, vector embeddings, and contextual reasoning paths, ensuring high-level AI agentic behaviors inherit the exact same capability protections as the low-level system [1.10, 1.20].

## Section VIII: Glossary for Non-Technical Readers

*   **System-on-Chip (SoC):** A single physical computer chip that combines all the essential components of a computer (like the central processor, graphics accelerator, memory controller, and caches) onto a single silicon die.
*   **Processor (CPU/GPU/NPU):** The physical execution units on a chip. The CPU acts as the general-purpose coordinator, the GPU acts as a high-throughput parallel math engine, and the NPU acts as a dedicated, low-power accelerator for running pre-trained artificial intelligence models.
*   **Memory Wall:** The physical performance bottleneck caused by the fact that processors calculate data thousands of times faster than memory chips (RAM) can retrieve or store it, leaving the processor waiting idly for data.
*   **Semantic Wall and Semantic Void [Empirical Horizon]:** The logical gap created when high-level safety rules (like array boundaries and memory limits) are completely erased during translation into raw, blind machine code. This leaves "voids" or un-monitored shadows in memory where security exploits can run unchecked.
*   **Cache Line:** The small, fixed blocks (typically 64 or 128 bytes) in which a processor's fast local cache retrieves data from slower main system memory (RAM).
*   **Serialization and Deserialization:** The slow, CPU-intensive process of converting structured objects in memory into flat text/bytes (like text files) to save them to a disk, and vice-versa.
*   **QnLang [Empirical Horizon]:** The proposed systems programming language designed to run capability-secure operations natively.
*   **cdqnOS [Empirical Horizon]:** The proposed lightweight operating system written natively in QnLang [Empirical Horizon].
*   **Physical Birth Register (PBR) [Empirical Horizon]:** The physical attestation record of a Qn primitive's birth, constructed by counting reference clock cycles elapsed between successive physical hardware noise events [1.23, 1.77].
*   **Causal Lineage Hash:** A cryptographic hash chain linking the newly born Qn primitive to its parent primitive's states, proving its structural heritage [1.10].
*   **QnPQC [Empirical Horizon]:** Speculative Post-Quantum Cryptographic Agility constructed as high-dimensional lattice manifolds of 1D Qn primitives at the compiler/OS layers, guaranteeing unforgeable, sovereign attestation [1.10, 1.15].
