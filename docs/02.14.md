### Abstract
This paper establishes the fourteenth step in the CDQN-02 series by proposing the mathematical, systems, and binary specification of the **Consolidated $QnIR_{\text{stable}}$ Bytecode Contract** for the **Stage 4 Intermediate Representation ($QnIR_4$)** [2, 10]. ## I. Introduction and the Bootstrapping Invariant

To construct a high-level programming language capable of expressing exact, multi-scale algebraic computations, we propose to verify each compiler ascension stage iteratively using physical use-case validation gates [1, 2].

### A. The Bootstrapping and Self-Hosting Invariants (The 2-Source Rule)
1.  **Alfred V. Aho, Monica S. Lam, Ravi Sethi, & Jeffrey D. Ullman (2006).** *Compilers: Principles, Techniques, and Tools* [23]. This classic text establishes that to prevent compilation-stage desynchronization and bootstrapping cycles when writing a self-hosting compiler, the intermediate representation (IR) must be locked under a stable, highly structured, and immutable contract.
2.  **Niklaus Wirth (1995).** *A Plea for Lean Software* [38]. This paper proves that restricting and simplifying compiler targets to a compact, non-redundant instruction-set minimizes compiler-stage technical debt and prevents execution-trace bloat during self-hosting.

### B. Defining the Target Options and Specification Bounds
To evaluate the portability and modularity of the intermediate representation across different computing substrates, the cdqn full-stack architecture defines three distinct target execution options [02.01.md]:
1.  **Option A (Sovereign Native):** Direct bare-metal execution on physical open-source hardware (such as RISC-V with hardware-enforced CHERI capability registers) running the custom **cdqnK** microkernel [02.01.md, 02.06.md].
2.  **Option B (Sovereign Virtual):** Speculative bare-metal execution on custom sovereign virtual machine environments (deferred to future epochs) [02.01.md].
3.  **Option C (Legacy Sandbox):** Portable sandboxed guest execution on un-hardened legacy processors (x86-64, ARM) using standard WebAssembly Text ($Wat_{\text{safe}}$) and WASI (WebAssembly System Interface) Preview 1 streams [02.01.md, 02.03.md].

By separating the intermediate representation's stable contract from these physical execution options, we hypothesize that the compiler's backend remains highly modular and adaptable, allowing the same $QnIR_{\text{stable}}$ bytecode to target either high-performance bare-metal silicon (Option A) or portable legacy sandboxes (Option C) [02.01.md].

## II. The $QnIR_{\text{stable}}$ Instruction Set Architecture (vISA)

We specify the formal instruction matrices, types, and binary encoding rules of the stable contract, ensuring that the target remains completely target-independent and compile-time verifiable [1, 2, 10].

### A. The Object and Wire Specifications
In accordance with the "Blind Register" and "Boundary-Level Verification" invariants of Chapter 1.9 and 2.02, all scale and proof metadata is erased at the register boundary [17, 02.02.md]. Wires are defined strictly as finite lists of un-scaled primitive types [10]:

$$\text{Obj}(\mathcal{C}) = \text{List}(\mathcal{T}) \quad \text{where} \quad \mathcal{T} = \{ \text{i32}, \; \text{i64} \}$$

The Zoom scale $z \in \mathbb{N}$ of the profinite truncation is restricted strictly to compile-time **Phantom Types**, checked and enforced statically by the compiler's type-checker before code generation and fully erased during the proof-erasure pass, keeping the hardware registers blind and maximizing ALU frequency [3, 54].

To execute scale-dependent page transitions under the **Spectral Sequence of Scales ($E_r$)** [39, 52], the required morphism restriction maps ($\rho_{z_1, z_2}$) are **statically injected as explicit, un-scaled integer-scaling boxes** (such as bit-shifting and padding multipliers) into the monoidal graph before the scale metadata is erased [2, 6, 26]. The physical CPU registers execute these scaling boxes blindly with zero runtime scale-tracking or metadata-resolution overhead [24, 29].

## III. The Memory-Slab Allocator and Capability Interface

We specify how the stable contract coordinates memory allocations and capability boundaries securely on physical silicon [3, 9, 14].

### A. The $p$-adic Tree-Structured Slab Allocator
Grounded in F. Q. Gouvêa (2020) and Martin Hofmann (1999), we compartmentalize the flat WebAssembly linear memory heap into disjoint, nested **$p$-adic memory balls** to prevent out-of-bounds corruption [8, 15]:
1.  **The Memory Layout:** The linear memory heap is partitioned into hardware-aligned memory slabs [02.06.md].
2.  **The In-Place Overwrite:** The stable `overwrite` instruction executes a constant-time in-place memory write at the exact same physical offset, bypassing the non-deterministic latencies of garbage collection [15, 32].
3.  **Thread-Local Stalk Isolation:** The allocator assigns each CPU core a private, independent memory sub-ball, eliminating "cache-coherence storms" on multi-core substrates [33].

### B. The Opaque Capability-Wrapped Handles (`externref`)
To prevent temporal exploits (such as use-after-free or dangling-pointer attacks), **the stable contract completely forbids exposing raw integer pointers to the guest program [1, 14].** 

All heap-allocated memory stalks and resources are wrapped strictly inside opaque, uninterpreted WebAssembly reference handles: **`externref`** [1, 14]:
*   The guest code is physically restricted from performing pointer arithmetic or casting handles to raw integer pointers [1, 14].
*   When executing bare-metal under `cdqnK` (Option A), these `externref` handles map directly to **hardware-enforced CHERI capabilities**, triggering an immediate CPU trap upon address-forgery or out-of-bounds memory access [14, 18].

## IV. Speculative Execution and Transactional Recovery Invariants

We specify how the stable contract coordinates speculative execution and transactional rollbacks to prevent microarchitectural timing and memory-safety exploits [3, 9, 29].

### A. The Speculative Thread-Join Barrier
Grounded in H. T. Kung & John T. Robinson (1981) and Santiago Arranz Olmos et al. (2025), the compiler backend statically injects **speculative barriers** (`FENCE` / `CSDB` on ARM, or conditional masking on RISC-V) at the `spec_split` and thread-join boundaries [30, 35]:
1.  **The Parallel Execution:** The speculative 2-adic thread (Core 1) executes the 2-adic integer component at maximum hardware frequency, completely unblocked by the 5-adic software emulation [`CDQN-02.06a`].
2.  **The Secondary Thread:** The 5-adic cofactor wire $A_5$ is passed as an opaque capability-handle to a background thread spawned on Core 2, executing multi-precision rational arithmetic as an asynchronous software task [32].
3.  **The Speculative Barrier:** The primary core is physically blocked from executing speculatively ahead until the asynchronous 5-adic remainder calculation on Core 2 is completed and verified [30, 35].

### B. Constant-Time Frame Scrubbing and Speculative Rollbacks
Grounded in Liyan Huang et al. (2025) and Hector Garcia-Molina & Kenneth Salem (1987), the Speculative Rollback Engine is designed to prevent data-dependent timing side-channel leaks [29, 34]:
1.  **The spec_rollback Opcode:** If a speculative transaction fails the background verification checks, the active Capability Supervisor immediately invokes the `spec_rollback` instruction [19, 34].
2.  **Constant-Time Frame Scrubbing:** The `spec_rollback` instruction is compiled to always execute the `$exit_with_scrub` routine, zeroing out the entire pre-allocated stack frame in a constant, data-oblivious number of clock cycles, completely neutralizing rollback timing side-channels [29, 34].
3.  **Linear-Typed State Isolation:** Because the memory-register operations are strictly linear/affine-typed, the compiler statically guarantees that a remainder pointer is consumed **exactly once**, mathematically restricting use-after-free, double-free, and re-entrant rollback exploits [3, 9, 32].

## V. The Advanced Scaling, Attestation, and Database Interfaces

To ensure the cdqn stack remains a quiet, zero-dependency, and non-bloated tool, we specify how the stable contract manages advanced hardware-scaling and database queries [4, 19, 25].

### A. The SPIR-V Functor and GPU Cohomology
Grounded in the Khronos Group SPIR-V Specification (2024), the compiler's backend maps our abstract monoidal parallel wires ($A \otimes B$) and localized sheaf gluing ($H^1 = 0$) directly into coordinate-free SPIR-V compute shaders [4]:
1.  **The Thread Mapping:** Monoidal tensor products ($\otimes$) compile directly into independent GPU/NPU compute thread IDs, executing with near-zero CPU thread-scheduling overhead [4, 25].
2.  **The GPU Cohomology:** The localized restriction maps $\rho$ are compiled directly into a Cellular Chain Complex of sparse boundary matrices represented as typed **SPIR-V Matrix Types** (`OpTypeMatrix`) [4, 41]. The physical GPU/NPU executes the linear programming check ($d^1 \circ d^0 = 0$) in parallel across thousands of independent execution lanes, bypassing CPU-GPU memory copy bottlenecks [4, 42].

### B. The `wasi-hardware` Attestation Interface
Grounded in Han Ge & Stephen R. Tate (2007) and Ernie Brickell et al. (2004), the `get_daa_sig` instruction is compiled to generate identity-hiding, post-quantum secure group signatures [20, 22]:
1.  **Bilinear Pairings:** The attestation is evaluated via bilinear pairing equations ($e: G_1 \times G_2 \to G_T$) over the Wasm table [20, 21].
2.  **Fuzzy Extractor DRAM Sweeps:** The hardware attestation is stabilized via Fuzzy Extractor DRAM sweeps, performing a bitwise AND with a static mask to stabilize the raw noisy entropy [19, 5].

### C. The Unified Hypermedia Sheaf (UHS) Data Model
Grounded in David I. Spivak & Bartholoměj Fong (2018), functorial data migration allows data schemas to map compatibly to different contexts [45]:
1.  **Database Persistence:** The database is compiled directly as a directory of linked, standard-compliant **Google Open Knowledge Format (OKF) v0.1** Markdown files, with no separate database engine or SQL query parser [50].
2.  **Relational Retrieval (`pullback`):** The structured query is compiled into a sequential, stack-bound directory walk using standard WASI I/O imports (`fd_read` / `fd_write` sealed in Wasm Table 0) [1, 24]. The parser walks the relative Markdown links and parses the YAML frontmatter on-the-fly, pushing matching stalks directly onto Wasm's native stack registers [1, 24, 45].
3.  **User Interface (HTML/DSD):** To project the updated sheaf back to the user, the Wasm compiler outputs a direct serialization routine [48, 49]. The routine reads the updated in-memory stalks and writes them directly into raw, declarative HTML markup containing Declarative Shadow DOM (DSD) components, allowing the browser's native C++ engine to render the interface instantly [48, 49].

### D. The Non-Destructive Algebraic Bypass Event
Grounded in Peter Selinger (2010) and Neil D. Jones et al. (1993), the compiler short-circuits arithmetic pipelines when dynamic variables resolve to absolute identities ($Q_0, Q_1$) [26, 51]:
*   **Static Constant Folding:** If an operand resolves statically to $Q_0$, the compiler rewrites the SMC graph, bypassing the `mul2` function completely and materializing `i64.const 0` directly on the stack [1, 26, 51].
*   **Dynamic Token Propagation:** If the operand resolves dynamically at runtime, the `cdqnK` scheduler intercepts the `mul2` dispatch, writes $Q_0$ directly to the output wire, and immediately propagates this **activation token** downstream to prevent parallel deadlocks [25]. The pending thread of $b$ is cancelled, and the runtime immediately executes $b$'s affine destructor ($!$), preventing memory and space leaks [9, 32].

## VIII. Glossary of Terms for Non-Technical Readers

We define the foundational concepts used in this specification to ensure interdisciplinary readability [1].

### A. Substructural and Allocator Terminology
*   **Symmetric Monoidal Category (SMC):** A mathematical structure where calculations are represented as diagrams of parallel wires (representing data states) and boxes (representing operations), allowing parallel and sequential steps to be compiled without a traditional program counter.
*   **Sovereign Self-Hosting:** The ability of a programming language to compile its own compiler source code without relying on legacy compilers or intermediate host runtimes.
*   **Host-Level / Kernel-Level Affine Teardown:** A security recovery protocol where the host operating system or kernel catches guest-level traps, walks the capability tables, and manually zeroes out speculative memory segments before recycling the sandbox to prevent data-remanence leaks.

### B. Capability and Licensing Terminology
*   **Regulation Sheaf ($\mathcal{R}_{\text{reg}}$):** The type-checking constraint in QnLang that mathematically monitors and enforces software license compliance (deontic logic) at compile-time, ensuring copyleft protection.
*   **Algebraic Effect:** A mathematical method of modeling computation side-effects (like I/O or GPU dispatches) as formal algebraic operations in a free monad, separating the definition of the action from its physical execution.
*   **Wasm Table:** A sandboxed WebAssembly memory array that stores function references securely, preventing guest programs from faking system-calls or executing unauthorized background processes.
