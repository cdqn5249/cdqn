### **1. Rationale for Revision**

This plan supersedes `plan_0000.md`. The initial implementation phase successfully proved the viability of the `cdqn` kernel logic. However, it also revealed a philosophical conflict between the manifesto's goal of a "stronger, simpler core" and the pragmatic use of large, complex, third-party dependencies for core runtime functions (e.g., persistence and networking). As detailed in `check_0001.md`, the decision has been made to pivot towards a development strategy that prioritizes **architectural sovereignty and minimalism** to create a truly bespoke and optimized runtime, faithful to the original vision.

### **2. New Guiding Principles**

*   **Maximum Sovereignty, Minimum Dependencies**: We will build all core runtime services (persistence, networking, scheduling) from scratch using only the Rust standard library wherever feasible.
*   **The "One Exception" Rule**: We will not "roll our own crypto" or our own binary serialization format. We will permit the use of a minimal, carefully vetted set of external libraries for these low-level, mathematically-focused, and universally audited tasks. The approved exceptions are:
    *   `serde` (for compile-time derive macros)
    *   `bincode` (for efficient, sovereign binary serialization)
    *   `sha2`, `ed25519-dalek`, `rand`, `hex` (for cryptographic primitives)
*   **Build, Don't Integrate**: The primary activity is to *build* bespoke components that are perfectly tailored to the `cdqn` workload, not to *integrate* generic, third-party frameworks.
*   **Security-First & TDD**: These principles remain paramount, especially as we take on the responsibility of writing more foundational code.

### **3. Revised Implementation Plan: The Sovereign Core**

#### **Phase 0: The Sovereign Baseline (Reset & Refactor)**

*   **Goal**: To strip the project of all non-essential dependencies and re-establish a clean, minimal, and verifiable baseline for the kernel.
*   **Key Milestones**:
    1.  **Purge Dependencies**: Edit `Cargo.toml` to remove `rocksdb`, `tokio`, `quinn`, `chrono`, and `serde_json`. Add `bincode`.
    2.  **Remove Obsolete Code**: Delete the `src/runtime/` directory and all code related to the `RocksDB`-based `C.Persistence` module.
    3.  **Re-establish Kernel Baseline**: Rewrite `main.rs` to perform the same in-memory KDU creation and verification that we achieved at the end of the original Phase 0. This ensures the `cdqnK` services function correctly on the new, minimal dependency set.
    4.  **Achieve Green CI Build**: Commit all changes and ensure the entire CI pipeline passes. This clean state is the official starting point for all new development.
*   **Exit Criteria**: The project compiles and runs successfully with the minimal dependency set. The CI pipeline is all green.

---

#### **Phase 1: The Sovereign Persistence Layer**

*   **Goal**: To build our own `C.Persistence` module from scratch, writing directly to the filesystem in a simple, robust, and log-structured format.
*   **Key Milestones**:
    1.  **Define the KDU Journal Format**: Design a simple binary format for storing KDUs on disk. The chosen format will be: `[8-byte length of KDU in little-endian format][Bincode-serialized KDU]`.
    2.  **Implement the `JournalWriter`**: Create the `C.Persistence` service with a `write_kdu` function that serializes a KDU using `bincode` and appends it to a journal file (e.g., `00000001.journal`).
    3.  **Implement Indexing**: Create a simple, in-memory `HashMap<String, u64>` that maps a `kdu_id` to its byte offset in the journal file. This index will be rebuilt by scanning the journal file(s) once upon node startup.
    4.  **Implement the `JournalReader`**: Implement a `read_kdu` function that uses the index to seek directly to the correct byte offset in the journal file, read the length prefix, and then deserialize the KDU.
    5.  **Integration and Verification**: Update `main.rs` to use the new, sovereign `C.Persistence` module to write a KDU, read it back via the index, and verify its integrity.
*   **Exit Criteria**: The custom-built persistence layer can successfully write and retrieve KDUs, and the entire process is verified by the CI pipeline.

---

#### **Phase 2: The Sovereign Networking Layer**

*   **Goal**: To build our own `cdqNetwork` module for peer-to-peer communication using only the Rust standard library's TCP sockets.
*   **Key Milestones**:
    1.  **Define the Wire Protocol**: Design the on-the-wire format for KDU transmission. For simplicity and consistency, it will be identical to the journal format: `[8-byte length prefix][Bincode-serialized KDU]`.
    2.  **Implement the `NodeServer`**: Create a service that can bind to a TCP port (`std::net::TcpListener`) and listen for incoming connections in a dedicated thread.
    3.  **Implement the `NodeClient`**: Create a service that can establish an outbound connection to a peer (`std::net::TcpStream`).
    4.  **Implement KDU Transmission Logic**: Write the functions to handle the precise sending and receiving of the length-prefixed KDU data over a TCP stream.
    5.  **Integration and Verification**: Update `main.rs` to accept command-line arguments to launch in either "server" or "client" mode. The test will involve launching two instances of the runtime locally, having the client connect to the server, and successfully transmitting a single KDU.
*   **Exit Criteria**: Two local instances of the `cdqnRuntime` can successfully exchange a KDU using the custom-built, standard-library-only networking stack.
