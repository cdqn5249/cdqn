## II. Mathematical Formulation of Algebraic Effects as Monads

To perform rigorous, compositional security analysis over system-level dispatches, we model computational side-effects purely category-theoretically [13, 27].

### A. Side-Effects as Free Monadic Operations
Grounded in Gordon D. Plotkin & John Power (2002) and Eugenio Moggi (1991), computational side-effects are modeled strictly as algebraic operations in a free monad over a signature $\Sigma$ [13, 27].

Let $\Sigma$ be a signature of operations representing our system capabilities (such as `wasi_io` or `spirv_compute`) [15]. We define the free monad $M(X)$ representing the computation space as [13, 27]:

$$M(X) \cong X + \sum_{\text{op} \in \Sigma} M(X)^{\text{Ar}(\text{op})}$$

Where:
*   $\text{Ar}(\text{op})$ represents the arity of the abstract operation (the type of its return parameters) [15].
*   The morphism $f: A \to M(B)$ represents a computation that can return a value of type $B$ or perform a sequence of abstract algebraic operations from the signature $\Sigma$ before returning [13, 27].

This monadic representation separates the *definition* of the side-effect from its *execution* [13]. The guest program does not perform the I/O; it simply yields an abstract, typed operation to the surrounding context [26].

### B. Effect Handlers as Algebra Morphisms
Grounded in Gordon D. Plotkin & Matija Pretnar (2009), algebraic effect handlers are formulated strictly as algebra morphisms that map the abstract monadic operations to concrete, target-specific implementation steps [26].

An effect handler $H$ for a monad $M(X)$ is defined as an algebra morphism [26]:

$$H: M(X) \to Y$$

satisfying the compatibility equations for the operations in the signature $\Sigma$.

The handler intercepts the yielded operation at the boundary [26]. If a program performs an abstract `write_char` operation, the handler catches the dispatch, reads the data, and translates it into a target-specific physical instruction (such as a standard WASI `fd_write` syscall under Option C, or a CHERI register write under Option A) [02.03.md, 26]. 

This guarantees that the guest code has zero direct access to the host's filesystems, registers, or network interfaces, enforcing absolute sandboxed modularity [1, 26].

## III. The WebAssembly Table-Based Capability Shield

We specify the compiler's backend lowering passes, demonstrating how the abstract algebraic effect handlers are translated into secure, table-based WebAssembly instructions [1, 2].

### A. Compiling Handlers to Isolated WebAssembly Tables
To prevent guest programs from faking system-calls or executing unauthorized background processes, the compiler's backend lowers the abstract algebra morphisms strictly to WebAssembly’s native, sandboxed table structures [1, 19]:
1.  **Table Isolation:** The compiler declares a private, immutable WebAssembly Table (`table`) of size 1, designated exclusively to store the verified host-level system-call function reference [1, 23]:
    ```wat
    (table 1 funcref)
    ```
2.  **Explicit Registration:** During the module's instantiation, the active Capability Supervisor registers the host-level system call (such as the WASI `fd_write` import) directly into the first slot of the table [19, 20]. The guest bytecode cannot read or write to this table, protecting the function reference from corruption [19, 65].

### B. Enforcing Software-Enforced CHERI Capability Safeguards
Grounded in the cWAMR capability-based WebAssembly runtime specifications (Subramanyan, 2025), all system-level dispatches are compiled strictly as **Table-Lookup `call_indirect` instructions** [19, 65]:
1.  **The Handle Check:** When the guest program performs an abstract operation, the compiler generates an index pointer representing the capability handle [19].
2.  **The Indirect Call:** The parser executes the indirect call, verifying the index natively on the stack [1]:
    ```wat
    (call_indirect (type $wasi_write_sig) (i32.const 0))
    ```
3.  **The Trapping Guard:** If the compiler or the runtime detects an instruction attempting to execute a system call outside of the declared table index (representing an unauthorized capability escalation) [65], **the virtual machine immediately traps**, aborting the guest execution instantly to prevent out-of-bounds corruption [1, 19].

When executing bare-metal under `cdqnK` (Option A), these table-lookup indices map directly to **hardware-enforced CHERI capability registers** [36 in 02.06]. The CPU's capability unit performs real-time, hardware-level tag and bounds verification. Any attempt to forge an address or reuse an expired capability clears the hardware tag bit, triggering an immediate CPU trap [36, 37 in 02.06].

## IV. Information-Flow Analysis and Branchless Compilation

At the physical gate level, the local processor must execute operations without exposing private data to microarchitectural state leaks [29, 30]. We specify how the compiler statically identifies and sanitizes secret-dependent control-flow paths before code generation [15].

### A. Static Information-Flow Control and Constant-Time Type-Tagging
Grounded in the information-flow control (IFC) lattices of SecWasm (Basys et al., 2022), the $QnLang$ compiler's frontend statically tracks the security provenance of every data wire [15, 67]:
1.  **Type Annotations:** When a developer declares a sensitive wire or morphism inside $QnLang_2$, they apply the **`@constant_time`** capability attribute (e.g. `shalt : i64_z -> 1 @constant_time`) [03.01.md].
2.  **Security Tagging:** The compiler's parser reads this attribute and tags the wire's entry in the Symbol Table with a strict `Secret` security label, while un-annotated wires are tagged as `Public` [15].
3.  **Transitive Propagation:** During semantic analysis, the compiler executes an information-flow propagation pass over the Symmetric Monoidal Category (SMC) hypergraph [50]. It transitively propagates the `Secret` label down the entire dependency chain [50]. If any variable carrying a `Secret` label is used as an input to a loop boundary, a conditional jump (`if`), or a memory address offset (which would leak state through the cache), **the compiler immediately blocks compilation with a strict security type exception [15, 50, 67].**

### B. Lowering Secret-Dependent Wires to Branch-Free `select` Instructions
Once the information-flow security lattice is validated, the backend JIT-compiler uses the security labels to select the safe microarchitectural execution paths [2, 9]:
1.  **Public Branching:** Wires carrying the `Public` label (such as the stack pointer `$sp` in `$exit_with_scrub` or the capability handle in `$perform_wasi_io`) are allowed to use standard, structured conditional branch instructions (`if/else` or conditional loops) [1, 2]. To prevent speculative execution bypasses (Spectre-style), the compiler statically injects **speculative barriers** (`FENCE` / `CSDB`) immediately after these public branches [30, 66].
2.  **Secret Selection:** Conversely, any data-dependent choice operating on a wire carrying the `Secret` label (such as the Soft Halt check `shalt`) is **strictly and exclusively compiled to WebAssembly's native branch-free `select` instruction [2, 29].** This flattens the execution flow. The host JIT-engine lowers the `select` directly to hardware-level conditional moves (`CMOVcc`), completely bypassing the CPU's branch predictor unit and ensuring that physical registers execute secret calculations in a constant, data-oblivious number of clock cycles [2, 29].

## VII. Glossary of Terms for Non-Technical Readers

We define the foundational concepts used in this specification to ensure interdisciplinary readability [1].

### A. Algebraic Effect Terminology
*   **Algebraic Effect:** A mathematical method of modeling computation side-effects (like I/O or GPU dispatches) as formal algebraic operations in a free monad, separating the definition of the action from its physical execution.
*   **Effect Handler:** A modular compiler mapping (algebra morphism) that intercepts abstract algebraic operations at the boundary and translates them into secure, target-specific physical instructions.
*   **Wasm Table:** A sandboxed WebAssembly memory array that stores function references securely, preventing guest programs from faking system-calls or executing unauthorized background processes.

### B. Compilation and Capability Terminology
*   **Single-Writer Wire:** A security constraint where every data channel has exactly one producer and one consumer, preventing data-race hazards across parallel processor cores.
*   **Host-Level Affine Teardown:** A security recovery protocol where the host operating system or kernel catches guest-level traps, walks the capability tables, and manually zeroes out speculative memory segments before recycling the sandbox to prevent data-remanence leaks.
*   **Speculative Barrier:** A hardware-level CPU instruction (such as `CSDB` or `FENCE`) that physically blocks speculative execution past a synchronization point, neutralizing Spectre-style cache side-channels.
