### Abstract
This paper establishes the second specification paper in the CDQN-03 series, designated as `03.02.md` [2]. Building upon the Minimal Seed Language syntax developed in Chapter 3.01, we address the challenge of managing dynamic heap allocations (such as lazy-evaluated remainder thunks and persistent ledgers) without the runtime overhead and non-deterministic latencies of garbage collection [1, 9]. We define the syntactic rules and compile-time verification passes for the linear (`-o`) and affine (`~`) type modalities [3, 15]. 

We specify the Single-Read Constraint, demonstrating how the compiler's type-checker statically verifies that linear resources are consumed exactly once, permitting safe in-place memory updates on the target-specific substructural allocator [9, 14, 18]. Furthermore, we formalize the compile-time synthesis of affine destructors, enabling the safe reclamation and scrubbing of registers during speculative thread rollbacks [15, 16]. This substructural type system serves as the resource-management mechanism of the high-level language.

## I. Introduction and Continuity from Chapter 3.01

To construct a high-level programming language capable of expressing exact, multi-scale algebraic computations [4], we must evaluate the optimal architecture for resource and memory management [1, 2].

### A. The Substructural Memory Challenge in Stage 1
In the Stage 0 Seed Language ($QnLang_0$), memory management was bypassed by restricting execution strictly to Wasm's native typed stack, eliminating the linear memory heap entirely from the compilation target [2]. While this stack-only constraint provides complete structural immunity to memory-corruption exploits, the transition to Stage 1 requires managing dynamic heap variables, such as carried algebraic remainders ($r_z = x - x_z$) and persistent database ledgers [14]. 

Traditional languages manage heap memory using either unmanaged pointers (which expose the system to buffer overflows, pointer-aliasing, and use-after-free exploits) or automatic garbage collectors (which consume CPU clock cycles, generate heat, and trigger physical thermal throttling on resource-constrained hardware) [1, 34, 35]. To resolve these limitations, we propose to govern the Stage 1 memory allocator using a **Substructural Type System** [3].

### B. Defining the Target Options and Specification Bounds
To evaluate the portability and modularity of these substructural types across different computing substrates, the cdqn full-stack architecture defines three distinct target execution options [02.01.md]:
1.  **Option A (Sovereign Native):** Direct bare-metal execution on physical open-source hardware (such as RISC-V with hardware-enforced CHERI capability registers) running the custom **cdqnK** microkernel [02.01.md, 02.06.md].
2.  **Option B (Sovereign Virtual):** Speculative bare-metal execution on custom sovereign virtual machine environments (deferred to future epochs) [02.01.md].
3.  **Option C (Legacy Sandbox):** Portable sandboxed guest execution on un-hardened legacy processors (x86-64, ARM) using standard WebAssembly Text ($Wat_{\text{safe}}$) and WASI (WebAssembly System Interface) Preview 1 streams [02.01.md, 02.03.md].

By separating the language's syntax from these physical execution options, we hypothesize that the compiler's frontend remains completely target-independent, allowing the same high-level source code to target either high-performance bare-metal silicon (Option A) or portable legacy sandboxes (Option C) [02.01.md].

## II. Mathematical Formulation of the Substructural Type System

To perform exact, zero-leak memory management at the linguistic boundary, we formulate our type system under the algebraic laws of Substructural Logics [3].

### A. The Strict Linear Type Invariant
Following the foundations of Jean-Yves Girard's linear logic [3] and Philip Wadler’s linear type theory [9], we restrict the algebraic remainder $r_z$ to be a linear resource type, denoted using the linear implication operator:

$$r_z : Q(S_z) \multimap Q(S_{z+1})$$

The core invariant of a linear type is that it must be consumed **exactly once** [9]. The type-checker statically verifies that the variable is neither duplicated (aliased) nor discarded silently without being formally evaluated [9]. This single-owner pointer constraint mathematically guarantees the absence of use-after-free exploits, pointer-aliasing, and spatial memory leaks on **the target-specific substructural allocator (such as the $p$-adic Tree-Structured Slab Allocator under QnIR_1) [9, 14, 18]**.

### B. The Affine Exception and Destructor Mapping
While linear types enforce strict resource conservation, they introduce a significant systems-programming failure under exceptional conditions:

*   **The Linear Exception Failure:** If a running thread panics or traps (due to a hardware fault, division-by-zero, or an asynchronous speculative thread rollback [02.06a.md]), the linear contract is violated. Because the active linear variable is never formally consumed, the compiler cannot generate deallocation instructions, leaving un-evaluated thunk pointers in the execution trace and generating memory leaks or data remanence [15, 17].
*   **The Required Solution:** We must upgrade the memory-handling logic from strictly linear to **affine** [15]. We define the affine modality (`~`) and synthesize affine exception recovery destructors to actively zero out variables upon thread termination.

An affine type requires that a resource must be consumed **at most once** [15]. If a thread panics or traps, **the active Capability Supervisor (the `cdqnK` kernel under Option A, or the WASI runtime host under Option C)** catches the trap signal, walks the instance's capability table, and forces the execution of the host-side affine destructors to actively zero out the memory-isolated stack frames and physical registers, preventing data-remanence leaks [15, 17].

## III. Syntactic Annotations and Multi-Wire Declarations

We specify the symbolic, math-aligned syntax of $QnLang_1$ used to declare substructural wire lifetimes on standard virtual keyboards [1].

### A. Syntactic Lifetime Modifiers
We completely eliminate verbose English keywords, representing wire lifetimes using explicit **substructural type modifiers** applied uniformly to the type signature to preserve the left-arrow (`<-`) binding invariant [1, 28]:
1.  **Linear Wires (`-o`):** Used to bind active remainder thunks and resource wires that must be consumed exactly once [9]:
    ```qn
    input_a <- -o i64_z   -- Linear wire; must be consumed exactly once
    ```
2.  **Affine Wires (`~`):** Used to bind optional or disposable resources that can be consumed at most once, permitting safe, explicit destruction via the `$discard` operator [15]:
    ```qn
    input_b <- ~ i64_z    -- Affine wire; can be explicitly discarded
    ```
3.  **Unrestricted Wires (`!`):** Used to bind read-only, static constants (such as Ring 0 physical constants) that can be freely copied via the exponential modality [3]:
    ```qn
    input_c <- ! i64_z    -- Unrestricted wire; can be duplicated
    ```

### B. Multi-Wire Lifetime Declarations
To eliminate typographical redundancy and reduce visual noise, $QnLang_1$ supports **Multi-Wire Declarations**, allowing multiple independent wires sharing the same lifetime modality and base type to be declared in a single, unified line [57, 69 in 03.01].

```qn
-- Declaring multiple linear and affine wires under strict modalities
balance_a, balance_b <- -o i64_z
temp_remainder       <- ~ i64_z
```

Because there are no behavioral classes, dynamic reference graphs, or pointers, this flat wire binding natively enforces **Data-Oriented Programming (DOP)** at the syntactic level, separating code from state [52 in 03.01].

## IV. Compile-Time Lifetime Validation Passes

We specify the compiler's semantic analysis passes used to verify substructural lifetimes before lowering the AST to the $QnIR_1$ bytecode [2, 10].

### A. Statically Verifying the Single-Read Constraint
To satisfy the strict linear and affine resource boundaries, the compiler’s type-checker implements a **Single-Read Constraint** [3, 9]:
1.  **The Compilation Pass:** The compiler's parser reads the sequential tokens of a monoidal pipeline (`input_a >> mul2 >> shalt`) [03.01.md].
2.  **The Reference Counter:** It counts the active read operations of each declared wire **(the target-specific read instructions, such as `local.get` in WebAssembly or register loads in physical ISAs)** [23 in 02.05].
3.  **The Validation:** If a linear variable (`-o`) is read multiple times (indicating pointer-aliasing) or left unread (indicating a resource leak), **the compiler blocks the compilation immediately with a strict type-safety error**, preventing memory corruption at the syntactic boundary [3, 9, 15].

```
                     SINGLE-READ LIFETIME CHECK
    [ QnLang_1 Source Code ] ──► Compiles: input_a >> mul2 >> shalt
                                               │
                                               ▼ (Parser scans reference reads)
                        Are linear constraints satisfied?
                        - Linear (-o): Reads == 1 ?   Yes
                        - Affine (~):  Reads <= 1 ?   Yes
                                               │
                                 ┌─────────────┴─────────────┐
                                 ▼ Yes                       ▼ No
                        [ COMPILE SUCCESSFUL ]      [ COMPILE TYPE ERROR ]
                        - Lowers to QnIR bytecode   - Aborts compilation
```

### B. Enforcing the Symmetric Monoidal Stack Invariant
To compile the high-level program into a strict Symmetric Monoidal Category (SMC) graph with zero-overhead register allocation, the type-checker enforces the **Symmetric Monoidal Stack Invariant** [2, 10]:
1.  **The Stack Alignment:** Because $QnLang$ utilizes **Point-Free Programming**, computations are represented as the direct composition of morphisms [03.01.md]. The type-checker verifies that the input and output stack heights match the function signatures exactly [2, 16 in 02.03].
2.  **Automatic Destructor Insertion:** If an affine variable (`~`) is left unconsumed at a branch termination point, the compiler's type-checker automatically injects **an explicit deallocation morphism (the `$discard` operator)**, lowering it to target-specific register-clearing instructions **(such as the `$discard_i64` function in $Wat_{\text{safe}}$ bytecode or raw masking instructions in physical ISAs)** [2, 15].

## VI. Glossary of Terms for Non-Technical Readers

We define the foundational concepts used in this specification to ensure interdisciplinary readability [1].

### A. Substructural Type Terminology
*   **Linear Type:** A specialized type-system rule where every variable (resource) must be used exactly once. It prevents data from being copied or deleted without the system's explicit knowledge, eliminating spatial memory leaks.
*   **Affine Type:** A modification of linear typing where a resource must be used at most once, permitting safe destruction and register scrubbing under exception handling.
*   **Unrestricted Type:** A standard data type that carries no lifetime restrictions, allowing variables to be copied or discarded freely.

### B. Allocation and Compiler Terminology
*   **Single-Read Constraint:** A compile-time check enforcing that every local variable representing a linear resource must be read exactly once, preventing use-after-free and double-allocation exploits.
*   **Point-Free Programming:** A programming paradigm where computations are expressed entirely through the direct composition of functions without referencing intermediate variable names, reducing compiler parsing complexity.
*   **Symmetric Monoidal Stack Invariant:** A compile-time constraint enforcing that the physical number of execution wires entering and leaving a computation box matches the WebAssembly stack signature exactly.
*   **Option A (Sovereign Native Execution):** The target execution path where the compiled program is lowered directly to physical open-source machine instructions (like RISC-V with hardware-enforced CHERI capability registers) running natively on bare-metal under the sovereign `cdqnK` microkernel [02.01.md, 02.06.md, 14, 18].
*   **Option B (Sovereign Virtual Execution):** A speculative target execution path where the compiled program runs on bare-metal sovereign virtual machine environments designed in future co-design epochs [02.01.md].
*   **Option C (Legacy Sandboxed Execution):** The portable target execution path where the compiled program is lowered to standard, stack-only WebAssembly Text ($Wat_{\text{safe}}$) and executed inside sandboxed legacy runtimes (like Wasmtime) on standard operating systems (Windows, Linux, macOS) [02.01.md, 02.03.md].
*   **Capability Supervisor:** The system-level boundary manager (the `cdqnK` kernel in Option A, or the WASI/Wasm runtime host in Option C) that intercepts guest-level faults or traps, walks the capability tables, and reclaims allocated resources [02.03.md, 14, 18].
