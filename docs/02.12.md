### Abstract
This paper establishes the twelfth step in the CDQN-02 series by proposing the mathematical, systems, and transactional specification of the **Saga-Based Topological Rollback Engine** for the **Stage 3 Intermediate Representation ($QnIR_3$)** [4, 10]. Building upon the SPIR-V parallel compilation functor developed in Chapter 2.11, we address the challenge of executing and reversing parallel, speculative resource transactions across decentralized networks without the latency and deadlock hazards of global locking protocols [13, 14 in 01.9]. 

We specify the speculative commit architecture under Optimistic Paging and formalize the compile-time generation of compensating Saga transactions [13, 36]. To satisfy the timing-safe and re-entrancy-free constraints of the local execution core, we specify a constant-time frame scrubbing protocol to eliminate rollback timing side-channels, and utilize linear type invariants to prevent double-reclamation of the $p$-adic slab memory [9, 16, 29]. This transactional recovery model serves as the state-reconciliation mechanism of the cdqn compiler co-design.

## II. Transactional Specifications of the Saga Engine

We specify the transactional and category-theoretic mechanics of how the Saga engine compiles speculative commits and compensating transactions [13, 36].

### A. Speculative Commits under Optimistic Paging
Grounded in H. T. Kung & John T. Robinson (1981) and Fred B. Schneider (1990), optimistic concurrency control allows transactions to execute speculatively to maximize throughput [73, 76]:
1.  **The Speculative Commit:** When a local transaction completes register execution, the local kernel immediately writes the computed state $x_{t+1}$ to a transient, speculative local buffer, *assuming* the homological invariants are satisfied [01.9.md, 73, 76].
2.  **Asynchronous Verification:** The heavy, distributed sheaf-theoretic gluing check ($H^1=0$) and physical hardware attestation are dispatched and executed asynchronously in the background, completely unblocking the primary register execution lanes [01.9.md, 02.11.md].

### B. Compiling Compensating Transactions (Sagas)
Grounded in Hector Garcia-Molina & Kenneth Salem (1987) and Gerhard Weikum & Gottfried Vossen (2001), distributed transactions are partitioned into a sequence of local transactions, where any failure triggers compensating actions to restore consistency [36, 75]:
1.  **The Compounding Chain:** The compiler represents the speculative pipeline as a sequence of Sagas, where each step has a corresponding, compiler-synthesized compensating transaction [36].
2.  **The Speculative Abort:** If the background sheaf check fails ($H^1 \neq 0$, indicating a factual exploit or double-allocation attempt) [01.14.md], the `cdqnK` scheduler immediately cancels the speculative pipeline and invokes the compensating Saga [02.06a.md, 36].

## III. Microarchitectural Speculative Security Invariants

We specify the microarchitectural register-scrubbing and memory-isolation invariants required to execute the speculative rollbacks securely on physical silicon, neutralizing speculative timing and re-entrancy exploits [9, 16, 29].

### A. Constant-Time Frame Scrubbing and Timing-Leak Mitigation
*   **The Derivation Chain:**
    *   *Initial Observation:* The loop-based memory-zeroing of the `$exit_with_scrub` function generates varying execution times depending on the stack depth [02.05.md, 02.07.md].
    *   *Empirical Sources:* Liyan Huang, Junzhou He, Chao Wang, & Weihang Wang (2025) [29] and Lehmann et al. (2021) [34] establish that variable-time execution loops are vulnerable to timing side-channels.
    *   *Structural Failure:* If the execution time of the speculative rollback loop varies based on the size of the unconsumed remainder ($r_z$), an attacker can measure the precise rollback latency (the timing of the abort) to reconstruct the secret remainder state, bypassing the branchless `#constant_time` selection gates [29].
    *   *Required Solution:* Implement **Constant-Time Frame Scrubbing** inside the `$exit_with_scrub` routine [29].

To flatten the execution latency during rollbacks, the allocator pre-allocates a fixed-size Slab A frame (e.g. exactly 125 elements, up to offset 1016) [02.07.md]. 

The `$exit_with_scrub` function is compiled to **always zero out the entire pre-allocated memory frame**, regardless of how many stack slots were actually used during the speculative run [02.07.md, 29]. 

The physical CPU executes the memory scrubbing in a fixed, constant number of clock cycles, completely neutralizing timing side-channels natively on the physical gates [29].

### B. Linear-Typed State Isolation and Re-entrancy Prevention
Grounded in Philip Wadler (1990) and Jesse A. Tov & Riccardo Pucella (2011), substructural type systems statically eliminate re-entrancy and double-free vulnerabilities [9, 16]:
1.  **The Linear Invariant:** Every speculative remainder thunk and memory pointer is typed strictly as a single-owner linear resource [02.06.md].
2.  **The Single-Consumption Constraint:** The compiler's type-checker statically proves that a linear variable can be consumed **exactly once** [9, 15]. 
3.  Because the type system structurally prevents the creation of duplicate pointer aliases, **it is mathematically impossible to trigger a double-reclamation of the $p$-adic slab memory or a re-entrant rollback call**, neutralizing the logical exploit surface at the syntactic boundary [3, 9, 15, 16].

## VI. Glossary of Terms for Non-Technical Readers

We define the foundational concepts used in this specification to ensure interdisciplinary readability [1].

### A. Transactional and Database Terminology
*   **Optimistic Paging:** A distributed concurrency control technique where transactions are speculatively written to local buffers under the assumption of success, while verification runs asynchronously in the background.
*   **Saga Pattern (Compensating Transaction):** A distributed transactional pattern that partitions a long-running transaction into a series of local steps, executing a sequence of "destructor" actions to reverse the completed steps if a failure occurs.
*   **Two-Phase Commit (2PC):** A legacy, high-overhead distributed database protocol requiring all nodes to lock their local registers and vote before committing, creating massive synchronization bottlenecks.

### B. Substructural and Security Terminology
*   **Host-Level / Kernel-Level Affine Teardown:** A security recovery protocol where the host operating system or kernel catches guest-level traps, walks the capability tables, and manually zeroes out speculative memory segments before recycling the sandbox to prevent data-remanence leaks.
*   **Symmetric Monoidal Stack Invariant:** A compile-time constraint enforcing that every stack register representing a linear resource must be read exactly once, preventing memory leaks and use-after-free exploits.
*   **$p$-adic Memory Ball:** A mathematically isolated memory compartment inside Wasm linear memory. Its boundaries are determined by $p$-adic metric equations, ensuring allocations are disjoint.
