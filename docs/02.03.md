### Abstract
This paper establishes the third specification paper in the CDQN-02 series, designated as `02.03.md` [2]. Building upon the strict, non-Cartesian Symmetric Monoidal Category (SMC) formalization developed in Chapter 2.02, we address the physical and microarchitectural frictions of compiling category-theoretic dataflow onto standard von Neumann hardware registers [3, 10]. We propose to map our abstract monoidal wire types (SMC types) and morphisms (boxes) to the WebAssembly Text Format ($Wat_{\text{safe}}$) using a strictly typed, memory-free, and stack-only target layout, strictly honoring the principles of Data-Oriented Programming (DOP) [1, 2, 34, 48]. 

We specify the monoidal wire-to-stack mapping, representing parallel wires as typed function parameters and local variables, and formalize the morphism-to-function lowering pass [2]. Furthermore, we construct the symmetry compiler-pass to translate coordinate-free wire crossings (`sym`) into safe, local register-swapping sequences on Wasm's typed execution stack [10, 26]. Finally, we enforce the Single-Writer Wire invariant by implementing the Symmetric Monoidal Stack Invariant, mathematically restricting pointer aliasing, use-after-free, and data-race exploits natively at compile-time [9, 50, 51].

## I. Introduction and Continuity from Chapter 2.02

In Chapter 2.02 (CDQN-02.02, v1.0.2), we formalized the core coordinate-free syntax of the $QnIR_0$ Stage 0 seed as a free strict Symmetric Monoidal Category ($\mathcal{C}$) [10]. We established the mathematical "laws of the graph," proving that vertical and horizontal compositions preserve strict algebraic coherence, and restricted the category to be strictly non-Cartesian (forbidding the diagonal map $\Delta_A$ and terminal map $!_A$) to enforce linear resource-conservation [9, 10, 51].

However, this abstract category-theoretic formulation introduces a subsequent systems-programming boundary:

*   **The Semantic Gap:** Abstract monoidal wire networks and coordinate-free string diagrams cannot be executed directly by physical silicon. They must be lowered to a concrete hardware instruction set architecture (ISA) [48].
*   **The Legacy Debt Risk:** If we map our category directly to physical assembly, we must write custom machine generators that handle hardware registers, memory boundaries, and operating system calls from day-one, trapping the compiler in platform-specific technical debt and unmanaged memory-safety leaks [1, 48].

To bridge this semantic gap, we propose the **$Wat_{\text{safe}}$ Stack-Only Execution Mapping** [2, 48]. By compiling $QnIR_0$ to the strictly typed, memory-free, and stack-only subset of the WebAssembly Text Format ($Wat_{\text{safe}}$), we establish an exploratory, secure compilation target that executes natively on standard silicon with near-hardware speeds, completely bypassing the dynamic memory-safety leaks of legacy compilers [2, 34].

### A. The Semantic Gap and Low-Level Realization
The semantic gap between abstract category theory and physical machine code is a documented software engineering friction [48]. Direct lowering to physical registers forces the compilation pipeline to handle low-level optimization complexities—such as register-allocation and platform-specific memory boundaries—early in the design cycle [3]. By targeting a virtual, stack-based Instruction Set Architecture (vISA), we isolate the core compiler logic, allowing the system to run on a highly structured, mathematically tractable, and target-independent dataflow model before physical code generation [2, 3].

### B. The Selection of the $Wat_{\text{safe}}$ Sandbox
Standard WebAssembly is often treated as a secure sandbox, but its unmanaged linear memory heap (`memory`) remains vulnerable to internal memory-corruption exploits (such as stack buffer overflows or use-after-free bugs) [34]. To eliminate these vulnerabilities, $QnIR_0$ targets strictly the **$Wat_{\text{safe}}$ subset** [2, 34]. 
*   By omitting the linear memory heap entirely from the compilation target, we restrict our program to execute exclusively on Wasm's native, typed operand stack [2].
*   The data cannot be corrupted because there is no user-addressable heap memory to overwrite, providing a mathematically secure and zero-debt environment for the Stage 0 seed [2, 34].

## II. The Monoidal Wire-to-Stack Mapping

To compile our strict Symmetric Monoidal Category ($\mathcal{C}$) to $Wat_{\text{safe}}$ natively, we must first map our abstract monoidal wire types to the WebAssembly execution stack, strictly separating our data from behavior to avoid the technical debt associated with Object-Oriented Programming (OOP) [1, 2, 10, 48]. 

Under the core principles of Data-Oriented Programming (DOP), we represent our data as flat, immutable values, entirely separate from any behavioral classes or dynamic reference graphs [1, 48]. Category-theoretic objects are not programming objects; they are abstract monoidal wire types representing primitive data channels [10].

### A. SMC Wires as Scaled Phantom Types
In Chapter 2.02, we defined the objects (monoidal wire types) of $\mathcal{C}$ as finite lists of scaled types representing parallel execution wires, where each wire is explicitly parameterized with its Zoom scale: $(t, z) \in \mathcal{T} \times \mathbb{N}$ [10]. 

Because the physical WebAssembly stack cannot natively track these scale properties at runtime, we propose to compile the scale $z$ strictly as a **Phantom Type** [6, 42]. `[Empirical Horizon]`
*   **The Compile-Time Check:** The scale $z$ is statically checked and enforced by the $QnLang$ compiler's type-checker to satisfy the Scale-Alignment Monad's invariants before code generation [6, 42].
*   **The Erased Lowering:** During the lowering pass to $QnIR_0$, the scale metadata is erased [6]. 

This guarantees that the Wasm stack-only local variables (`local`) representing our wires remain completely free of runtime type-tagging or scale metadata bloat, while preserving compile-time scale safety [2, 6]:

```wat
(local $wire_1 i64)
(local $wire_2 i64)
```

Because WebAssembly local variables are strictly typed and managed directly by the host runtime's register-allocator, this mapping eliminates the need for our Stage 0 compiler to perform manual CPU register allocation, maintaining a highly elegant and lightweight compiler design [2, 3].

### B. Monoidal Wires as Parameter Concatenation
The parallel combination of two wire-systems $A \otimes B$ is represented as the concatenation of their typed lists (Chapter 2.02) [10]. Under the $Wat_{\text{safe}}$ mapping, this parallel composition is compiled as the **concatenation of typed parameter lists** [2, 24].

```
                  MONOIDAL WIRE-TO-PARAMETER MAP
    [ Abstract SMC Object: A ⊗ B ]
                 │
                 ▼ (Concatenates wire-lists: [i64] ⊗ [i64] -> [i64, i64])
    [ Wasm Typed Parameter List ]
                 (param i64 i64)
```

Let $f: A \to B$ be a categorical operation where the input monoidal wires are $A = [\text{i64}, \text{i64}]$ and the output is $B = [\text{i64}]$. The parallel input wires are compiled directly into the Wasm parameter signature:

```wat
(param i64 i64) (result i64)
```

This guarantees that the physical number of execution wires entering and leaving a computation box matches the Wasm stack signature exactly, enforcing structural type-safety at the compiler level [2, 10].

## III. The Morphism-to-Function Mapping

We specify how the abstract, localized computation boxes (morphisms) and their composition laws are compiled into structured, stack-based WebAssembly instructions [2, 10].

### A. Category Boxes as Structured Wasm Functions
Every morphism $f: A \to B$ in $QnIR_0$ is compiled directly into a native, statically defined WebAssembly function [2, 10]:

```wat
(func $morphism_f (param i64 i64) (result i64)
  ;; Local compilation of the box logic
)
```

We propose that this function-based representation enforces strict modularity and isolation. The internal execution logic of the box $f$ is completely encapsulated inside the function boundary, and it can only interact with the system by consuming the parameters pushed onto the stack and returning the result [2, 10].

### B. Horizontal Composition as Function Chaining
Let $f: A \to B$ and $g: B \to C$ be two sequential morphisms in $\mathcal{C}$, compiled to functions `$f` and `$g` respectively. We compile their **horizontal composition ($g \circ f$)** as a sequential chain of static function calls [2, 10]:

```wat
;; Compile horizontal composition g o f
call $f
call $g
```

```
               HORIZONTAL COMPOSITION LOWERING
    [ SMC Morphism: g o f ] ───► Compile: call $f \n call $g
```

Under WebAssembly's execution semantics, the function `$f` consumes its parameters from the typed stack and pushes its output ($B$) back onto the stack [2]. The subsequent instruction `call $g` immediately consumes $B$ as its input parameter, pushing the final result ($C$) onto the stack [2]. 

We hypothesize that this stack-based function chaining executes with zero memory allocation or temporary register spilling, maintaining native-speed execution without requiring complex runtime control-flow engines [2, 4]. `[Empirical Horizon]`

## IV. The Symmetry Compiler Pass and Algebraic Bypass Events

To represent the permuting and crossing of parallel wires without losing structural trackability, $QnIR_0$ must compile the abstract symmetry morphism `sym` and dynamic bypass loops into physical stack-manipulation instructions [10, 26].

### A. Compiling Wire Crossings to Stack-Swapping Sequences
Let $\text{sym}_{A, B}: A \otimes B \to B \otimes A$ be the symmetry morphism crossing two parallel wires of type `i64`. In our strict SMC, this is a pure geometric crossing [10, 26]. On a physical stack, however, we cannot cross wires geometrically. We must translate the crossing into stack-swapping instructions [2].

We propose to compile the `sym` operator strictly as an inline sequence of local variable read and write operations, utilizing the $Wat_{\text{safe}}$ stack registers [2, 10]:

```wat
;; Compile symmetry sym_A,B where A (local 0) and B (local 1)
local.get 0   ;; Push wire A onto the stack
local.get 1   ;; Push wire B onto the stack
local.set 0   ;; Pop B and write to local 0 (Wire A's original slot)
local.set 1   ;; Pop A and write to local 1 (Wire B's original slot)
```

Because this swap sequence operates entirely on Wasm's native stack registers, **no heap memory is allocated, and no pointers are generated** [2]. The wire crossing is executed securely with zero risk of use-after-free or stack-overflow exploits [2, 34].

### B. The Symmetry Elimination Optimization Pass
If a compiled string diagram contains consecutive, redundant wire crossings—such as executing `sym` followed immediately by its inverse—the compiler can optimize the graph before generating code [26].

We propose to implement a strict **Symmetry Elimination Pass** inside the compiler’s middle-end [22, 26]: `[Empirical Horizon]`

$$\text{sym}_{B, A} \circ \text{sym}_{A, B} \quad \longrightarrow \quad \text{id}_{A \otimes B}$$

*   **The Graph Rewrite:** The compiler scans the $QnIR$ monoidal graph [22].
*   **The Optimization:** If it detects a composition of two consecutive symmetry nodes, the compiler's graph-rewriter prunes both nodes from the active DAG on-the-fly, replacing them with a simple identity morphism [22, 26].
*   The compiler generates **zero machine instructions** for the crossed wires, completely bypassing the physical register-swapping and stack-shuffling penalty on legacy hardware (Option C) [3, 26].

### C. The Compile-Time Algebraic Bypass Event
When the compiler or the runtime JIT-engine detects an expression containing dynamic variables that resolve to absolute identities (such as multiplication by $Q_0$ or division by $Q_1$), the system must execute the **Algebraic Bypass Event** [3, 44]. `[Empirical Horizon]`

To execute this safely without generating execution-order dependency leaks, we map the Bypass Event to a compile-time **Constant Folding and Dead-Code Elimination (DCE)** pass [3, 62]:
*   **The Short-Circuit:** If the compiler detects `i64.const 0` as an active multiplication operand, it short-circuits the entire Wasm function chain, bypassing the physical ALU multiply instructions completely [3, 62].
*   **The Materialization:** It directly materializes the constant result `i64.const 0` into the output stack register, logging a `Bypass Event` in the compilation telemetry and immediately propagating the activation token to downstream functions to prevent deadlocks [43, 62].

## V. The Single-Writer Wire Invariant

In Chapter 1.9, we specified that every dataflow wire in the CDQN stack must behave as a **Single-Writer Wire** (a single-producer, single-consumer channel) to completely eliminate Write-After-Read (WAR) and Write-After-Write (WAW) data hazards across parallel multicore processors [50].

### A. Compiling Wires as Single-Assignment Channels
Under the $Wat_{\text{safe}}$ mapping, we propose to enforce this single-writer invariant by compiling every monoidal wire strictly as a **Single Static Assignment (SSA)** local variable [2, 50]. 
*   Every local variable representing a wire is assigned a value exactly once using a single `local.set` or `local.tee` instruction [2].
*   Once assigned, the variable is treated as immutable [50]. 
*   Because category-theoretic string diagrams contain no shared, mutable global states, **this single-assignment mapping is guaranteed by the structure of the category itself**, completely eliminating write-after-write (WAW) hazards natively on physical hardware [10, 50].

### B. The Single-Read Constraint and Trap Semantics
To satisfy the strict linear-logic resource boundaries established in Chapter 1.8, the compiler must guarantee that a resource is consumed exactly once [9]. 

We propose to implement the **Symmetric Monoidal Stack Invariant** within the compiler’s type-checker [2, 9]: `[Empirical Horizon]`
1.  **The Compilation Constraint:** Every Wasm local variable representing an active $QnIR$ wire must have **exactly one** read operation (`local.get`) in the generated output bytecode [2, 9].
2.  **Compile-Time Validation:** The compiler's validator scans the generated Wat text. If a local variable is read multiple times (indicating pointer aliasing or double-allocation), or if a variable is left unread (indicating a resource leak), **the validator blocks the compilation immediately with a strict type-safety error** [9, 51].
3.  **Wasm Trap Semantics:** If a thread panics or crashes (such as a hardware-level division by zero or a runtime assertion failure), WebAssembly's native execution engine immediately throws a fatal **trap**, aborting the call stack instantly without running guest-level destructors [2]. Because $Wat_{\text{safe}}$ executes exclusively on the host VM's native stack registers (with zero user-addressable heap memory), temporal safety on a fatal trap is securely guaranteed by the **host OS process teardown** and the Wasm VM's sandbox isolation [2]. The host OS automatically reclaims, scrubs, and zeroes out the native stack frame upon thread termination, preventing any data remanence or memory leaks without requiring slow guest-level heap-cleaning routines [2].

## VI. Glossary of Terms for Non-Technical Readers

*   **vISA (Virtual Instruction Set Architecture):** A standardized, platform-independent virtual machine language (such as WebAssembly) that acts as an intermediate target, allowing software to compile natively to any host processor without emulation.
*   **WebAssembly Text Format (Wat):** The human-readable, S-expression-based text representation of WebAssembly, allowing developers to write and audit Wasm code directly.
*   **$Wat_{\text{safe}}$ Subset:** A strictly typed, memory-free, and stack-only subset of the WebAssembly Text Format. By eliminating the use of the shared linear memory heap, it is structurally immune to memory-corruption exploits like buffer overflows.
*   **Algebraic Bypass Event:** A compiler optimization technique where a high-level logical operation resolving to absolute identities (such as multiplication by $Q_0$) is intercepted and short-circuited during compilation, allowing the execution flow to bypass the physical ALU and propagate the activation token downstream immediately.
*   **Single-Writer Wire:** A security constraint where every data channel has exactly one producer and one consumer, preventing data-race hazards across parallel processor cores.
*   **Symmetric Monoidal Stack Invariant:** A compile-time constraint enforcing that every stack register representing a linear resource must be read exactly once, preventing memory leaks and use-after-free exploits.
*   **Data-Oriented Programming (DOP):** A programming paradigm that separates code (behavior) from data (state), representing data as simple, immutable, and generic structures to maximize CPU cache performance and systems modularity.
