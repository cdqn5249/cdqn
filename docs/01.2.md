### Abstract

This document provides the formal architectural specification for the **Quang Number (Qn)**, succeeding the theoretical framework established in Paper 01.1. We define the Qn as a 64-bit computational primitive structured as a **Triad of Dependencies**: Magnitude ($m$), Narrative ($n$), and Logic ($l$). This specification details the bit-level partitioning, the mechanism of **Dimensional Lamination** [Empirical Horizon], and the implementation of a hardware-native **Resonance Predicate** [Empirical Horizon] for ontological security. We ground the Qn's state transitions—from fluid states to **Invariant Quang Numbers (iQn)**—in an analogical mapping of Graph Rigidity Theory to semantic dependency networks. Furthermore, we provide a quantitative performance model for "Time-Gain" and a strategy for custom dialect lowering via the MLIR infrastructure. This specification establishes the technical baseline for the `cdqnLang` compiler and the multi-dimensional execution host within the Chained and Distributed Quang Network.

### 1. The Container: 64-Bit Word Alignment and Parallelism

The Quang Number is designed for immediate integration into current semiconductor environments, utilizing the 64-bit word as its atomic container. This choice is predicated on maximizing throughput on existing register files while enabling multi-dimensional data density.

#### 1.1 Hardware-Native Throughput and Memory Trade-offs
Alignment to the 64-bit boundary ensures the Qn utilizes existing load/store units and cache line architectures without the "Abstraction Penalty" of non-standard bit-widths. 
*   **Empirical Support:** **Arm Ltd. (2025)** and **Intel Corporation (2025)** manuals confirm that 64-bit aligned data access is the prerequisite for peak memory bandwidth and atomic consistency. 
*   **Bandwidth Trade-off:** While a 64-bit Qn doubles the memory footprint of a standard 32-bit float, we posit that the **Instruction Count Reduction** achieved by bypassing iterative logic and software-level type-checking results in a net energy and temporal gain. **Hennessy & Patterson (2024)** argue that in the modern "Memory Wall" era, reducing the total number of required memory-to-logic roundtrips via higher data density is more efficient than minimizing raw bit-width.

#### 1.2 Tri-Dimensional Partitioning (The m:n:l Triad)
The 64-bit word is partitioned into three distinct functional segments, representing the **Lamination** [Empirical Horizon] of the Mathematical, Relational, and Behavioral dimensions. This partitioning allows the hardware to treat a single scalar move as a multi-dimensional state transfer.

| Bit Segment | Segment Name | Dimension | Role |
| :--- | :--- | :--- | :--- |
| **[0 - 31]** | **m** (Magnitude) | Mathematical | Logarithmic Quantitative Scale |
| **[32 - 47]** | **n** (Narrative) | Relational | Semantic Neighborhood / Prefetch Hint |
| **[48 - 63]** | **l** (Logic) | Behavioral | Ontological Mask / Capability Signature |

### 2. Segment `m`: The Mathematical Dimension (Magnitude)

The 32-bit Magnitude segment utilizes a **Logarithmic-Scaled Manifold** [Empirical Horizon] to bypass the iterative latencies of legacy IEEE 754 arithmetic.

#### 2.1 Logarithmic Arithmetic (LNS) Efficiency
The Magnitude segment treats values as logarithms, transforming multiplication and division into single-cycle additions and subtractions. This eliminates the high-latency iterative circuits (e.g., Newton-Raphson) traditionally required for division.
*   **Empirical Support:** **Arnold (2022)** and **Coleman et al. (2002)** demonstrate that Logarithmic Number Systems (LNS) provide constant relative precision and significantly lower power consumption for multiplication-heavy workloads. 
*   **Addition Approximation:** To address the historical complexity of LNS addition, the Qn Logic segment (`l`) provides pre-calculated bit-masks for the hardware **Resonance Mask**, approximating the logarithmic sum $log(A+B)$ via the techniques documented by **Frantz (2009)** for high-speed hardware implementation.

#### 2.2 Symbolic Exactness and Lazy Evaluation
To prevent "Rounding Drift," the Qn Magnitude maintains **Symbolic Ratios** [Empirical Horizon] within the log-space, utilizing a strategy of deferred execution to maintain absolute precision for rationals.
*   **Empirical Support:** **Boehm (2009)** provides the foundation for using lazy evaluation in exact arithmetic. By representing a number as a deferred operation graph rather than a truncated decimal, the Qn maintains 100% precision through the computational chain. Decimal materialization is treated as a secondary UI-layer event, preserving the mathematical integrity of the "Work" phase.

### 3. Segment `n`: The Relational Dimension (Narrative)

The `n` segment (Bits 32-47) serves as the **Semantic Address** [Empirical Horizon], identifying the contextual neighborhood and lineage of the primitive.

#### 3.1 Neighborhood-ID and Cooperative Prefetching
The 16-bit Narrative segment acts as a pointer to a **Neighborhood-ID** in the local CDQN registry. This ID facilitates hardware-software cooperative prefetching by identifying the "Relational Cluster" associated with the value.
*   **Empirical Support:** We extend the logic of **Ainsworth & Jones (2017)** regarding graph-aware prefetching and ground the hardware-software co-design in the framework provided by **Solihin (2020)**. The Narrative segment acts as a **Contextual Hint**; the hardware prefetcher uses the Narrative bits to load the Qn's "Chained Neighbors" into the cache before the execution unit requests them, significantly reducing memory-bound latencies.

#### 3.2 Quang Isotopes and Lineage Preservation
By separating Narrative from Magnitude, the system natively supports **Quang Isotopes** [Empirical Horizon]—primitives with identical magnitudes but distinct contextual ancestors. 
*   **Architectural Role:** This preserves **Data Provenance**, preventing the loss of metadata that occurs when context is flattened into raw binary. For example, a "Price" Isotope and a "Velocity" Isotope are mathematically equivalent but narratively isolated, ensuring that the machine respects the original intent of the data through every stage of processing.

### 4. Segment `l`: The Behavioral Dimension (Logic)

The `l` segment (Bits 48-63) is the **Hardware Capability Mask** [Empirical Horizon], enforcing ontological rules at the gate level.

#### 4.1 Invariant Behavioral State (IBS) Assignment
The Logic segment houses the **IBS-Hash** [Empirical Horizon]. This 16-bit signature represents an ontological behavior prototype that governs how the magnitude is allowed to interact.
*   **Assignment Methodology:** The first 1,024 hashes are **Static Ontological Prototypes** based on **Semantic Primes** (**Goddard & Wierzbicka, 2024**). The remaining space is reserved for **Context-Dependent Behaviors** assigned by the local CDQN registry. 
*   **Empirical Support:** **Levy (1984)** and **Miller et al. (2020)** provide the foundation for hardware-enforced capability systems, which the Qn extends from memory-safety to logical-safety at the level of the individual primitive.

#### 4.2 The Resonance Predicate $R(l_1, l_2)$
Interaction between Qns is gated by a bitwise compatibility check performed synchronously with the arithmetic operation.
*   **Predicate Logic:** $R(l_1, l_2) = (l_1 \text{ AND } l_2) \neq 0$. 
*   **Function:** $R$ is a hardware-level gate that verifies if two primitives share a compatible behavioral phase. If "Out of Phase" (e.g., attempting to add a "Scalar" to a "Logic Flag"), the operation is physically blocked at the register-write-back stage, providing **Native Ontological Security**.

### 5. Dimensional Lamination and Notation

A Qn is a **Projected State** [Empirical Horizon] where dimensions are bit-sliced into the 64-bit word to maximize execution speed.

#### 5.1 Bit-Level Parallelism
The Qn architecture utilizes **Bit-Slicing** to achieve synchronous tri-dimensional validation within a single instruction. 
*   **Empirical Support:** **Hennessy & Patterson (2024)** detail how bit-level parallelism and specialized bit-manipulation instructions (like the ARMv9 `SVE` or Intel `AVX-512` extensions) enable a single clock cycle to process multiple independent data planes. The Qn leverages this to perform Magnitude calculation ($m$) while simultaneously verifying Narrative proximity ($n$) and Logic resonance ($l$).

#### 5.2 The Simplified `m:n:l` Syntax
To ensure consistency across GitHub documentation and the `cdqnLang` compiler, we adopt a colon-separated format for representing the Qn in text.
*   **Notation:** `Magnitude:Narrative:Logic`
*   **Unit Representation:** `1:Self:Unit` (A base unit with Magnitude 1, self-referential narrative, and unit logic).
*   **Isotope Representation:** `0.5:Transparency:IBS_021` (Magnitude 0.5 in the context of visual transparency using the IBS_021 rule-set).

A Qn transitions from a "Fluid" state to a "Solid" state as its dependency network saturates, creating a "Ground Truth" for the network.

#### 6.2 The Saturation Threshold (The Rule of 4)
We apply **Jackson & Jordán (2005)** regarding rigidity matroids and **Erdős & Rényi (1960)** regarding connectivity thresholds in random graphs to determine when a Qn becomes unchangeable.
*   **Mechanism:** A Qn reaches the **Saturation Threshold** [Empirical Horizon] when it possesses **4 stable dependencies** to established iQn anchors. At this connectivity density ($k=4$), the probability of the semantic coordinate "drifting" (hallucinating) under a domain shift drops toward zero, rendering the node "Rigid."

Once a Qn is promoted to the Invariant phase, it is moved to the **Rigidity Map** [Empirical Horizon]—a protected memory heap managed via **Linux dma-buf** and **ARM Memory Tagging Extension (MTE)**. These iQns serve as the immutable foundation upon which all other fluid Qns in the CDQN are calculated.

### 7. Implementation: The cdqnLang Virtual ISA

The architecture is implemented as a Software-Defined Hardware layer using the **MLIR** infrastructure to ensure hardware-agnostic portability.

#### 7.1 Compiler Deployment and Lowering
The `cdqnLang` compiler utilizes a multi-stage lowering pipeline to transform high-level intent into optimized hardware kernels.
*   **MLIR Dialect:** We define a custom `qn` dialect to preserve the triad's semantics during optimization.
*   **Lowering Strategy:** Following the methodology of **Phipps et al. (2023)**, the `qn` dialect is lowered into SPIR-V or LLVM IR kernels. These kernels are specifically designed to utilize the Fused Multiply-Add (FMA) units of existing GPUs and NPUs, effectively "re-programming" the hardware's arithmetic units to process Qn logic.

#### 7.2 Performance Model: The Time-Gain Equation
To ground our performance claims, we utilize a model based on the reduction of iterative software overhead:
$$T_{gain} = I_{reduction} \times (C_{hit} - B_{overhead})$$
*   **$I_{reduction}$:** The factor of instruction count reduction gained by bypassing software-level type-checking and iterative division.
*   **$C_{hit}$:** The cache hit-rate increase derived from Narrative-based cooperative prefetching.
*   **$B_{overhead}$:** The bandwidth penalty of using 64-bit words for all operations.
*   **Verification:** Detailed benchmarking will be performed in the **03 Series Papers** via cycle-accurate **gem5** simulation.
