### Abstract

This document provides the formal architectural specification for the **Quang Number (Qn)**, succeeding the theoretical framework of Paper 01.1. We define the Qn as a 64-bit computational primitive structured as a **Triad of Dependencies**: Magnitude ($m$), Narrative ($n$), and Logic ($l$). This v1.6 refinement formalizes the **Strict Inheritance Axiom** [Empirical Horizon] for `cdqnLang` syntax, eliminating any syntactical ambiguity between Canonical Form (Long Notation) and Operator Shorthand (Short Notation). We mandate explicit declarative typing for all contextual and behavioral shifts. Furthermore, we formalize the **Typed Charge System** [Empirical Horizon], assigning **Bit [52]** as the **Operational Spin** (Charge) bit within the Logic segment, eliminating traditional sign bits from the mathematical manifold. We introduce the **Contextual Key** [Empirical Horizon] within the Narrative segment, specifying the bit-range for toggling between Subjective (Debt) and Objective (Charge) interpretation modes. Supported by dual-sourced architecture principles and verified compiler strategies, this specification establishes the technical baseline for the multi-dimensional execution host of the Chained and Distributed Quang Network.

### 1. The Container: 64-Bit Word Alignment and Parallelism

The Quang Number utilizes the 64-bit word as its atomic container to maximize throughput on existing register files while enabling multi-dimensional data density.

#### 1.1 Hardware-Native Throughput and Memory Alignment
Alignment to the 64-bit boundary ensures the Qn utilizes existing load/store units and cache line architectures without the "Abstraction Penalty" of non-standard bit-widths. 
*   **Empirical Support:** **Hennessy & Patterson (2024)** and **ARM Ltd. (2023)** manuals confirm that 64-bit aligned data access is the prerequisite for peak memory bandwidth, atomic consistency, and maximizing SIMD utilization in modern ARMv9-A and x86_64 architectures.
*   **Data Density Benefit:** While a 64-bit Qn increases individual fetch size, the **Instruction Count Reduction** achieved by bypassing iterative logic results in a net temporal gain. Increasing data density per fetch is more efficient than minimizing raw bit-width in the modern "Memory Wall" era.

#### 1.2 Tri-Dimensional Partitioning (The m:n:l Triad)
The 64-bit word is partitioned into three distinct functional segments, representing the **Lamination** [Empirical Horizon] of Mathematical, Relational, and Behavioral dimensions.

| Bit Segment | Segment Name | Dimension | Role |
| :--- | :--- | :--- | :--- |
| **[0 - 31]** | **m** (Magnitude) | Mathematical | Unsigned Logarithmic Scale |
| **[32 - 47]** | **n** (Narrative) | Relational | Contextual Key / Semantic Address |
| **[48 - 63]** | **l** (Logic) | Behavioral | Ontological Mask / Operational Spin |

### 2. Segment `m`: The Mathematical Dimension (Magnitude)

The 32-bit Magnitude segment bypasses the iterative latencies of legacy IEEE 754 arithmetic by utilizing an unsigned **Logarithmic-Scaled Manifold** [Empirical Horizon].

#### 2.1 Logarithmic Arithmetic (LNS) and Approximation
The Magnitude segment treats values as logarithms, transforming multiplication and division into single-cycle additions and subtractions.
*   **Empirical Support:** **Arnold (2022)** and **Coleman et al. (2002)** demonstrate that LNS significantly lowers power consumption and provides constant relative precision. 
*   **Addition Mitigation:** To address LNS addition complexity, the Logic segment provides pre-calculated bit-masks for hardware sum approximations. We ground these techniques in **Mitchell (1962)**, the canonical reference for logarithmic approximation, ensuring high-speed hardware feasibility.

#### 2.2 Symbolic Exactness and Unsigned Magnitude
The 32 bits of segment `m` are **Unsigned**, gaining dynamic range by delegating "Charge" to the Logic segment. Magnitude maintains **Symbolic Ratios**[Empirical Horizon] via deferred execution.
*   **Empirical Support:** **Boehm (2009)** provides the foundation for using lazy evaluation in exact arithmetic. By deferring decimal materialization until the UI boundary, the Qn maintains 100% precision for rationals during the computational "Work" phase.

### 3. Segment `n`: The Relational Dimension (Narrative)

The `n` segment (Bits 32-47) serves as the **Contextual Key** [Empirical Horizon], dictating the interpretation of the number's intent.

#### 3.1 Contextual Key Encoding
A reserved bit-range within the Narrative segment identifies the "Operational Mode" of the Qn.
*   **Encoding:** Bits[32-34] are allocated for Mode Selection.
    *   **Mode 0x0-0x3 (Subjective):** Human/Economic scale. Operational Spin (Bit 52) is interpreted as **Debt**, which restricts statistical choice.
    *   **Mode 0x4-0x7 (Objective):** Universal/Physics scale. Operational Spin is interpreted as **Charge**, enabling statistical choice.
*   **Hardware Impact:** We extend **Ainsworth & Jones (2017)** and **Solihin (2020)** regarding cooperative prefetching. The Narrative segment identifies "Relational Clusters," allowing the hardware to pre-load chained neighbors into cache before they are requested.

#### 3.2 Neighborhood-ID and Quang Isotopes
The Narrative segment acts as a pointer to a **Neighborhood-ID** in the local CDQN registry, enabling **Quang Isotopes** [Empirical Horizon]—primitives sharing magnitude but possessing distinct contextual ancestors.

### 4. Segment `l`: The Behavioral Dimension (Logic)

The `l` segment (Bits 48-63) is the **Hardware Capability Mask**[Empirical Horizon].

#### 4.1 Bit-Level Partitioning: IBS and Spin
The Logic segment is partitioned for high-speed gating:
1.  **Bits [48-51] (IBS Signature):** Identifies the **Invariant Behavioral State** (Void, Unity, Dual, etc.).
2.  **Bit [52] (Operational Spin):** Defines the "Charge" (0 = Presence; 1 = Reflected Intent).
3.  **Bits [53-63] (Permissive Mask):** Defines allowed operators (ADD, FUSE, etc.).

#### 4.2 The Resonance Predicate and Side-Channel Mitigation
Interaction is gated by a bitwise compatibility check.
*   **Precedence Rule:** The IBS-Type (bits 48-51) takes precedence over the Operational Spin (bit 52). Ontological compatibility is verified before sign logic.
*   **Side-Channel Defense:** Following **Kocher (1996)**, security-hardened implementations utilize **Constant-Time Execution** for spin detection, ensuring that the Logic-over-Spin check does not leak data through timing variance.
*   **Empirical Support:** **Levy (1984)** and **Miller et al. (2020)** ground this in capability-based security, enforcing safety at the register level.

### 5. Dimensional Lamination and Syntax Axioms

A Qn is a **Projected State**[Empirical Horizon] where dimensions are bit-sliced into the 64-bit word. To prevent syntactical ambiguity in the `cdqnLang` compiler, we enforce rigorous notation axioms.

#### 5.1 Bit-Slicing and Parallel Validation
The CDQN utilizes **Bit-Slicing** to achieve synchronous tri-dimensional validation. 
*   **Empirical Support:** **Hennessy & Patterson (2024)** detail how bit-level parallelism enables a single clock cycle to process multiple data planes. The Qn performs Magnitude calculation ($m$) while simultaneously verifying Narrative mode ($n$) and Logic resonance ($l$).

#### 5.2 Canonical Form (Long Notation)
When full explicit definition is required, `cdqnLang` mandates the Canonical Form.
*   **Syntax:** `Magnitude : Narrative : Logic` (or `m:n:l`).
*   **Example:** `0.5 : Transparency_Context : IBS_Dual`
*   **Requirement:** This notation **must** be used whenever custom metadata is injected or a Qn is newly instantiated at the UI/Application boundary.

#### 5.3 Operator Shorthand and the Strict Inheritance Axiom
When expressing invariant constants and their operations, the compiler supports a strictly governed shorthand (e.g., $Q_0, Q_1, Q_2$).

**1. The Absolute Ground State Maps:**
When an invariant is written in isolation, it strictly defines a hardcoded triad:
*   $Q_0 \equiv -\infty : \text{Origin} : \text{IBS-Void}$
*   $Q_1 \equiv 0 : \text{Origin} : \text{IBS-Unity}$
*   $Q_2 \equiv \log(2) : \text{Origin} : \text{IBS-Dual}$

**2. The Strict Inheritance Axiom [Empirical Horizon]:**
When an invariant is used as an operator against a variable subject (e.g., $A \times Q_2$), the shorthand invokes deterministic hardware-level lamination.
*   **Magnitude ($m$):** Computed via standard LNS ALU paths.
*   **Narrative ($n$):** Strictly defined as $n_A \oplus \text{Operator\_Event}$. The operator's absolute narrative (`Origin`) is erased; the subject's context ($n_A$) is inherited and appended with the interaction event.
*   **Logic ($l$):** Strictly defined by the evaluated Resonance Predicate $R(l_A, \text{IBS-Operator})$.

**3. The Prohibition of Implicit Override:**
*   **Rule:** No semantic ambiguity or "implied" masking is permitted. Grounded in the type-theoretic formalisms of **Pierce (2002)**, implicit coercions of structural metadata compromise type-safety.
*   **Enforcement:** If a developer intends for the resulting Qn to possess a Narrative or Logic state that differs *in any way* from the exact deterministic hardware lamination described above, the use of short notation is **strictly forbidden**. The `cdqnLang` compiler will throw a syntax fault for implicit metadata shifts. The developer *must* use Canonical Form (Long Notation) to explicitly declare the new $n$ and $l$ boundaries.
