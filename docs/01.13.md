## II. The Problem of Floating-Point Multi-Currency Ledgers

In classical financial accounting systems and distributed ledger networks (such as standard databases or blockchains), currency balances and exchange rates are represented using classical floating-point decimals (IEEE 754) or unconstrained fixed-point decimals.

### A. The Non-Associative Rounding Drift
As analyzed in Chapter 1.1, floating-point addition and multiplication are non-associative. When performing sequential currency conversions across asynchronous nodes, this non-associativity introduces microscopic rounding errors [4].

Let $A$, $B$, and $C$ be three distinct currency types (e.g., USD, EUR, VND). We evaluate the transaction of converting 1 unit of $A$ to $B$, then $B$ to $C$, versus converting $A$ directly to $C$. Under classical floating-point arithmetic, the sequential operations accumulate rounding drift:

$$(1.0 \times \text{Rate}_{A \to B}) \times \text{Rate}_{B \to C} \quad \neq \quad 1.0 \times \text{Rate}_{A \to C}$$

### B. The Arbitrage and SMR Vulnerability
Because of this rounding discrepancy, the financial ledger suffers from three system-level failures:
1.  **Consensus Splits:** Different database replicas executing identical transaction sequences in different thread-scheduling orders will accumulate divergent rounding states, violating the determinism constraint of State Machine Replication (SMR) [5].
2.  **Arbitrage Exploits:** Malicious actors can exploit these microscopic decimal discrepancies across asynchronous ledger partitions to execute automated "rounding attacks" (generating "free money" through recursive circular exchanges) [6].
3.  **Representation Loss:** Converting standard human-readable decimal inputs (such as $\$0.10$, which has an infinite repeating binary representation) to binary registers immediately introduces a representation error before any transaction is executed [4].

## III. Functorial Semantic Mapping of Exchange Rates

To prevent arbitrary rounding discrepancies and logical arbitrage exploits, we replace flat decimal exchange ratios with category-theoretic **Functorial Semantics** [1, 2, 3].

### A. The Currency Category ($\mathcal{S}$)
Following the frameworks of applied compositionality, we model the multi-currency system as a category of currency spaces $\mathcal{S}$, where:
*   Objects are currency types (USD, EUR, VND, etc.).
*   Morphisms $f: A \to B$ are the exact, verified exchange transaction pathways.

We define the mapping from the currency category $\mathcal{S}$ to the $Qn$ primitive category $\mathcal{C}$ as a **Functor** $\mathcal{F}$ [1, 2, 3]:

$$\mathcal{F}: \mathcal{S} \to \mathcal{C}$$

Because $\mathcal{F}$ is a functor, it is mathematically required to preserve identity and composition [2, 3]:

$$\mathcal{F}(\text{id}_A) = \text{id}_{\mathcal{F}(A)}$$

$$\mathcal{F}\big( A \xrightarrow{\;f\;} B \xrightarrow{\;g\;} C \big) \quad \equiv \quad \mathcal{F}\big( A \xrightarrow{\;g \circ f\;} C \big)$$

### B. Compile-Time Arbitrage Prevention
This functorial constraint is enforced statically by the $QnLang$ compiler.
*   When a financial smart contract is compiled, the type-checker verifies that the transaction path commutes.
*   If a transaction attempts an exchange path that violates this functorial composition (attempting to generate a decimal discrepancy), the compiler flags a *morphism composition violation* [1].
*   The transaction is blocked at the type-checking phase before any bytecode is generated, mathematically preventing rounding-arbitrage exploits by construction.

## IV. Coprime Base Decomposition and Remainder Carriage on Registers

To execute these exact currency transactions on standard binary hardware registers without introducing representation loss, the $Qn$ primitive performs exact base translations.

### A. The Algebraic Decomposition
The number 10 is composite, meaning the 10-adic absolute value does not define a multiplicative norm on its own. However, under the algebraic properties of profinite rings, the ring of $10$-adic integers $\mathbb{Z}_{10}$ decomposes exactly into the product of the $2$-adic and $5$-adic integers [7]:

$$\mathbb{Z}_{10} \cong \mathbb{Z}_2 \times \mathbb{Z}_5$$

By extending this to their fields of fractions, we obtain the coprime decomposition [7, 8]:

$$\mathbb{Q}_{10} \cong \mathbb{Q}_2 \times \mathbb{Q}_5$$

### B. The Execution Path
When a user inputs a decimal fraction (e.g., $0.1_{10} = 1/10$), the $QnLang$ compiler splits the value into two parallel registers:

$$0.1_{10} \mapsto \left( x_2, \; x_5 \right) \in \mathbb{Q}_2 \times \mathbb{Q}_5$$

The **$2$-adic component** ($x_2$) is executed natively on standard binary CPU registers for maximum processing speed. The **$5$-adic component** ($x_5$) represents the fractional portion that cannot be represented exactly in binary (such as the prime factor 5 in $\$0.10$). It is carried algebraically as an exact, symbolic cofactor (the remainder $r_z$) [8].

### C. Non-Destructive Remainder Carriage
During sequential execution, this remainder $r_z$ is managed by the compiler as a **single-owner linear type** ($\multimap$) [9]:

$$r_z : Q(S_z) \multimap Q(S_{z+1})$$

Because the linear type system guarantees that $r_z$ has exactly one active reference, the `cdqnK` kernel performs safe, in-place register updates [9]. When the precision of a balance is deepened during transaction processing, the kernel overwrites the register containing $r_z$ in-place:
*   The old thunk pointer is immediately consumed and deallocated in $O(1)$ constant time, preventing any space leaks or pointer-chase latency [9].
*   The Algebraic Simplification Filters (Schönhage Fast GCD) are executed immediately after every operation, reducing the fraction to its coprime normal form [10]:
    $$\text{FastGCD}(r_z) \to \frac{a}{b}$$
    restricting denominator expansion to a stable, linear growth rate $O(z)$ [10].

This guarantees that the physical register holds the exact, un-truncated mathematical value of the currency balance, completely eliminating representation drift.

## VI. Glossary of Terms for Non-Technical Readers

*   **Rounding Attack:** A financial exploit where a malicious actor takes advantage of tiny decimal rounding differences in a computer's arithmetic to generate un-backed currency through circular trades.
*   **Functorial Semantics:** A category-theoretic method of assigning meaning to a system, where the relationships between objects must be preserved exactly during translation.
*   **Morphism Composition:** The algebraic process of chaining two operations together (such as converting currency A to B, then B to C) into a single, unified operation (converting A directly to C).
*   **Coprime Base Decomposition:** Breaking down a composite number base (like base 10) into its prime, non-overlapping components (like base 2 and base 5) to allow exact, parallel calculations on computer registers.
*   **Linear Type:** A type system rule where every variable (resource) must be used exactly once, preventing data from being copied or deleted silently.
