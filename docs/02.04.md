## II. Constant-Time 2-adic Arithmetic Operators

To execute exact, multi-scale arithmetic natively on physical silicon without introducing timing vulnerabilities, the $QnIR_0$ bytecode specifies the $2$-adic addition and multiplication operators as flat, branch-free monoidal morphisms [2].

### A. The 2-adic Addition Operator
Let $A = [\text{i64}]$ and $B = [\text{i64}]$ be two un-scaled input wires representing our $2$-adic integer valuations on Wasm's operand stack (Chapter 2.03) [2]. We define the $2$-adic addition operator as the morphism [2]:

$$\text{add2}: A \otimes B \to A$$

Under the $Wat_{\text{safe}}$ mapping, `add2` is compiled directly into a native WebAssembly function that consumes the top two `i64` elements from the operand stack, executes the hardware-level 64-bit addition, and pushes the result back [2]:

```wat
;; Compile 2-adic addition morphism (add2: A ⊗ B -> A)
(func $add2 (param $x i64) (param $y i64) (result i64)
  local.get $x
  local.get $y
  i64.add
)
```

Because 64-bit integer addition `i64.add` is natively executed by the physical CPU ALU in a fixed, constant number of clock cycles on modern SoCs, it is structurally immune to data-dependent timing variations [2].

### B. The 2-adic Multiplication Operator
We define the $2$-adic multiplication operator as the morphism [2]:

$$\text{mul2}: A \otimes B \to A$$

Under the $Wat_{\text{safe}}$ mapping, `mul2` is compiled into a native WebAssembly function [2]:

```wat
;; Compile 2-adic multiplication morphism (mul2: A ⊗ B -> A)
(func $mul2 (param $x i64) (param $y i64) (result i64)
  local.get $x
  local.get $y
  i64.mul
)
```

On all modern physical hardware, 64-bit integer multiplication `i64.mul` is executed in constant-time [1]. Compiling these primitive operators as pure, stack-bound Wasm functions guarantees that the CPU executes arithmetic natively in a deterministic number of clock cycles, with zero heap-memory overhead and zero register-spilling latency [1, 2].

## III. The Branch-Free Soft Halt Node (`shalt`)

To determine exact convergence and terminate iterative loops on physical silicon without exposing the system to instruction-timing side-channels, we must resolve the Zero-Equivalence Problem of our projective remainders without branching [44].

### A. The Zero-Equivalence Derivation Chain
*   **Initial Observation:** Compiling the zero-equivalence check of the remainder ($r_z = 0$) into standard conditional structures (`if/else`) is necessary to yield CPU cycles back to the host [44].
*   **Empirical Sources:**
    1.  **Liyan Huang, Junzhou He, Chao Wang, & Weihang Wang (2025).** *WaSCR: A WebAssembly Instruction-Timing Side Channel Repairer* [63].
    2.  **Santiago Arranz Olmos, Gilles Barthe, Lionel Blatter, Benjamin Grégoire, & Vincent Laporte (2025).** *Preservation of Speculative Constant-time by Compilation* [64].
*   **Structural Failure:** Standard conditional branch instructions (such as `br_if` in Wasm or `JZ` in x86-64 assembly) are compiled by the host engine into hardware-level conditional jumps. These jumps trigger the CPU's branch predictor, creating **data-dependent timing variations** and exposing the execution thread to speculative execution side-channels (Spectre-style) [64]. An attacker can observe the timing differences between a zero and a non-zero remainder to reconstruct sensitive financial or coordinate states.
*   **Required Solution:** Eliminate the `if/else` block completely from the `shalt` node. The compiler must compile the Soft Halt as a **constant-time, branch-free selector**.

### B. Compiling the shalt Selector Node
We propose to compile the **Soft Halt** as a specialized, terminal string diagram node [2]:

$$\text{shalt}: A \to \mathbf{1}$$

where $\mathbf{1}$ represents the terminal unit object (WASI system exit) [23]. 

The `shalt` operator is compiled as a native $Wat_{\text{safe}}$ function that consumes the numerator $a$ of the remainder, executes a zero-comparison check, and uses Wasm's native, branch-free **`select`** instruction to determine the exit status before calling the WASI `proc_exit` function [2, 23]:

```wat
;; Compile Branch-Free Soft Halt node (shalt: A -> 1)
(func $shalt (param $numerator i64)
  (local $status i32)
  
  ;; Standard constant-time selection logic:
  ;; Stack order for select is [val1, val2, cond] where:
  ;; - val1 = value if cond is non-zero (numerator == 0 -> success status 0)
  ;; - val2 = value if cond is zero     (numerator != 0 -> failure status 1)
  i32.const 0       ;; val1 (value if cond is non-zero -> exit 0)
  i32.const 1       ;; val2 (value if cond is zero -> exit 1)
  local.get $numerator
  i64.eqz           ;; cond (pushes 1 if numerator == 0, else 0)
  select            ;; Pops condition, returns 0 if true, 1 if false (CMOVcc mapping)
  local.set $status
  
  local.get $status
  call $proc_exit
)
```

At the physical gate level, WebAssembly JIT engines (such as Wasmtime and V8) compile Wasm's `select` instruction directly to the hardware's constant-time conditional move instructions (`CMOVcc` on x86-64, or conditional mask-operations on ARM and RISC-V). 

Because both potential exit statuses are loaded onto the register file and selected via a hardware conditional move, **no conditional jumps are generated, the CPU's branch predictor is completely bypassed, and the execution time remains perfectly identical and flat-line** regardless of the remainder's value. This guarantees absolute resistance to instruction-timing side channels [63].

## IV. The Algebraic Bypass Event Interceptor

To prevent the compiler from generating redundant arithmetic pipeline instructions for dynamic variables that resolve to absolute identities ($Q_0$ or $Q_1$), $QnIR_0$ implements the **Algebraic Bypass Event Interceptor** [3, 44].

### A. Static Constant Folding and Short-Circuiting
We propose to map the Bypass Event to a compile-time **Constant Folding and Dead-Code Elimination (DCE)** pass [3, 62]. 

When the $QnLang$ compiler parses an algebraic expression, it evaluates the monoidal wires statically. If an operand resolves to $Q_0$ during compilation, the compiler executes a **Symmetric Monoidal Category Wire-Rewrite** [3]:
1.  **The Short-Circuit:** If the compiler detects `i64.const 0` as an active multiplication operand, it short-circuits the entire Wasm function chain, bypassing the physical ALU multiply instructions completely [3, 62].
2.  **The Materialization:** It directly materializes the constant result `i64.const 0` into the output stack register, logging a `Bypass Event` in the compilation telemetry [3, 62].

### B. Dynamic Token Propagation and Deadlock Prevention
If the calculation is dynamic (where the variables are computed in parallel across different cores), the Bypass Event is executed dynamically by the `cdqnK` scheduler [50].

```
                     THE DYNAMIC WIRE BYPASS
    [ Unresolved Wires: a, b, c ] ──> [ Addition Morphism: add3 ]
                                              │
                                              ▼ (Run-time valuations resolve: Q_1, Q_0, Q_0)
                                      Bypass Event Triggered!
                                              │
                                              ▼ (ALU pipeline bypassed)
    [ Write Q_1 directly to Output Wire ] ────┼───> Flag: [ Bypass Event ] logged
                                              │
                                              ▼ (Propagate Token Downstream)
    [ Downstream Morphism Executed Instantly ]
```

*   **Token-Passing Propagation:** When a thread returns a short-circuiting value (e.g., $a = Q_0$ in $a \times b$), the scheduler intercepts the `mul2` node [50]. It writes the determined value $Q_0$ directly to the output wire, and immediately propagates this **activation token** to the downstream functions [3].
*   **Preventing Deadlocks:** Because the output wire is populated immediately, the downstream threads waiting for this value can execute instantly [3, 50]. The system does not suffer from deadlock or synchronization stalls because the dataflow sequence is preserved [50].
*   **Resource Reclamation:** The pending thread of $b$ is cancelled, and the runtime immediately executes $b$'s **affine destructor** ($!$) to safely reclaim and zero out its linear-typed registers, preventing memory and space leaks [9, 51].

## VI. Glossary of Terms for Non-Technical Readers

*   **Soft Halt (`shalt`):** A natural, algebraic termination of a calculation when the exact mathematical result is reached (the remainder is zero). It stops the processor from wasting energy on redundant steps.
*   **Constant-Time Selector (`select`):** A compiler instruction that selects between two data values based on a condition without executing a conditional branch, bypassing the CPU's branch predictor and ensuring identical execution times regardless of the input data.
*   **Wasm Trap:** A fatal execution error thrown natively by the WebAssembly virtual machine, aborting the call stack instantly to prevent memory corruption.
*   **Algebraic Bypass Event:** A compiler optimization technique where a high-level logical operation resolving to absolute identities (such as multiplication by $Q_0$) is intercepted and short-circuited during compilation, allowing the execution flow to bypass the physical ALU and propagate the activation token downstream immediately.
