## II. The 2-adic Arithmetic Operators

To perform exact, zero-drift arithmetic on physical registers, the $QnIR_0$ bytecode specifies the $2$-adic addition and multiplication operators as strict monoidal morphisms [10, 26].

### A. The 2-adic Addition Operator
Let $A = [(\text{i64}, \; z)]$ and $B = [(\text{i64}, \; z)]$ be two scaled input wires at zoom level $z$ (Chapter 2.03) [10]. We define the $2$-adic addition operator as the morphism [10]:

$$\text{add2}: A \otimes B \to A$$

Under the $Wat_{\text{safe}}$ mapping, `add2` is compiled directly into a native WebAssembly function that consumes the top two `i64` elements from the operand stack, executes the hardware-level 64-bit addition, and pushes the result back onto the stack [2]:

```wat

(func $add2 (param $x i64) (param $y i64) (result i64)
  local.get $x
  local.get $y
  i64.add
)
```

Because the parameters are loaded directly from local registers and computed via the native CPU ALU, the execution runs with zero heap-memory overhead and zero register-spilling latency [2, 3].

## III. The Soft Halt Node and Zero-Equivalence

In Chapter 1.6, we established that the uncomputed remainder (the projective residual $r_z = x - x_z$) is carried algebraically as an exact, coprime rational fraction:

$$r_z = \frac{a}{b} \in \mathbb{Q}_p$$

Executing iterative loops or continuous calculations on legacy hardware requires a strict, energy-efficient mechanism to determine when a calculation has reached its exact, stable limit, allowing the processor to halt [43, 44].

### A. Decidability of the Zero-Equivalence Problem
The **Zero-Equivalence Problem** (proving whether a given symbolic expression is equal to zero) is generally undecidable in computer algebra [44]. 

However, because $Qn$ represents the remainder as an exact coprime fraction, the `cdqnK` kernel resolves the Zero-Equivalence Problem in $O(1)$ constant time [44]. 

A fraction $\frac{a}{b} = 0$ if and only if the numerator is exactly zero:

$$a = 0 \implies r_z = 0$$

By reducing the fraction to its canonical normal form, the system can determine exact convergence with a simple, constant-time integer comparison [43, 44].

### B. Compiling the shalt Termination Node
We propose to compile the **Soft Halt** as a specialized, terminal string diagram node [26]:

$$\text{shalt}: A \to \mathbf{1}$$

where $\mathbf{1}$ represents the terminal object (the WASI system exit) [23]. 

The `shalt` operator is compiled as a native $Wat_{\text{safe}}$ function that consumes the numerator $a$ of the remainder, executes a zero-comparison check, and if the remainder is exactly zero, immediately calls the WASI `proc_exit` function with a success status of `0` [2, 23]:

```wat
;; Compile Soft Halt node (shalt)
(func $shalt (param $numerator i64)
  local.get $numerator
  i64.eqz
  if
    i32.const 0   ;; Push success status 0 onto stack
    call $proc_exit
  end
)
```

If $r_z = 0$, the calculation has reached its absolute mathematical limit [44]. Continuing to execute the loop would result in redundant operations ($0 \oplus 0$), wasting CPU cycles and generating heat [33]. 

The `shalt` node immediately terminates the register execution loop and yields processing cycles back to the host, protecting the hardware's physical lifespan and achieving perfect thermodynamic stability [1, 33].

## IV. The Algebraic Bypass Event Interceptor

To prevent the compiler from generating redundant arithmetic pipeline instructions for dynamic variables that resolve to absolute identities ($Q_0$ or $Q_1$), $QnIR_0$ implements the **Algebraic Bypass Event Interceptor** [3, 44].

### A. Static Constant Folding and Short-Circuiting
We propose to map the Bypass Event to a compile-time **Constant Folding and Dead-Code Elimination (DCE)** pass [3, 62]. `[Empirical Horizon]` 

When the $QnLang$ compiler parses an algebraic expression, it evaluates the monoidal wires statically. If an operand resolves to $Q_0$ during compilation, the compiler executes a **Symmetric Monoidal Category Wire-Rewrite** [3, 10]:
1.  **The Short-Circuit:** If the compiler detects `i64.const 0` as an active multiplication operand, it short-circuits the entire Wasm function chain, bypassing the physical ALU multiply instructions completely [3, 62].
2.  **The Materialization:** It directly materializes the constant result `i64.const 0` into the output stack register, logging a `Bypass Event` in the compilation telemetry [3, 62].

### B. Dynamic Token Propagation and Deadlock Prevention
If the calculation is dynamic (where the variables are computed in parallel across different cores), the Bypass Event is executed dynamically by the `cdqnK` scheduler [49, 50]. `[Empirical Horizon]`

```
                     THE DYNAMIC WIRE BYPASS
    [ Unresolved Wires: a, b, c ] ──> [ Addition Morphism: add3 ]
                                              │
                                              ▼ (Run-time valuations resolve: Q_1, Q_0, Q_0)
                                      Bypass Event Triggered!
                                              │
                                              ▼ (ALU pipeline bypassed)
    [ Write Q_1 directly to Output Wire ] ────┼───> Flag: [ Bypass Event ] logged
                                              │
                                              ▼ (Propagate Token Downstream)
    [ Downstream Morphism Executed Instantly ]
```

*   **Token-Passing Propagation:** When a thread returns a short-circuiting value (e.g., $a = Q_0$ in $a \times b$), the scheduler intercepts the `mul2` node [50]. It writes the determined value $Q_0$ directly to the output wire, and immediately propagates this **activation token** to the downstream functions [3].
*   **Preventing Deadlocks:** Because the output wire is populated immediately, the downstream threads waiting for this value can execute instantly [3, 50]. The system does not suffer from deadlock or synchronization stalls because the dataflow sequence is preserved [50].
*   **Resource Reclamation:** The pending thread of $b$ is cancelled, and the runtime immediately executes $b$'s **affine destructor** ($!$) to safely reclaim and zero out its linear-typed registers, preventing memory and space leaks [9, 51].

## VI. Glossary of Terms for Non-Technical Readers

*   **Soft Halt (`shalt`):** A natural, algebraic termination of a calculation when the exact mathematical result is reached (the remainder is zero). It stops the processor from wasting energy on redundant steps.
*   **Hard Halt:** A forced termination of a calculation when the computer's physical memory or register limits are reached. It saves the uncompleted part of the calculation to be resumed later.
*   **Algebraic Bypass Event:** A compiler optimization technique where a high-level logical operation resolving to absolute identities (such as multiplication by $Q_0$) is intercepted and short-circuited during compilation, allowing the execution flow to bypass the physical ALU and propagate the activation token downstream immediately.
*   **Double-Allocation Exploit:** A system vulnerability where a single physical resource is allocated to two separate transactions simultaneously, leading to data corruption or financial theft.
*   **Wasm Trap:** A fatal execution error thrown natively by the WebAssembly virtual machine, aborting the call stack instantly to prevent memory corruption.
