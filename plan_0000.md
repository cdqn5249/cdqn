*   **ID**: `plan_0000.md`
*   **Date**: 19 September 2025
*   **Status**: Complete
*   **Vibe Coding Engine**: Gemini 2.5 Pro, Google

---

### **1. Guiding Principles**

*   **Phased & Iterative Development**: The ecosystem's complexity will be managed by implementing it in distinct, sequential phases. Each phase will deliver a testable, functional vertical slice of the architecture, building upon the previous one.
*   **Security-First Mindset**: Security is not a feature but a foundational requirement. Every component, from the kernel to the application layer, will be designed and built with an adversarial perspective. The CI/CD pipeline will include mandatory security audits from day one.
*   **Test-Driven Development (TDD)**: Core components, especially in the Kernel and Runtime, will be developed using TDD. A comprehensive test suite will be the ultimate specification for the system's correct behavior.
*   **Clarity Over Premature Optimization**: The initial focus will be on creating a clear, correct, and secure implementation of the architecture. Performance optimizations will be addressed systematically in a dedicated phase after the core functionality is proven.

---

### **2. Phase 0: The Sovereign Foundation (Single-Node MVP)**

*   **Goal**: To build a stable, single-node `cdqnRuntime` and the `cdqnLang` toolchain. This phase focuses on proving the core programming model and entity lifecycle before introducing network complexity.
*   **Key Milestones**:
    1.  **Repository & CI/CD Setup**:
        *   Initialize the `cdqn5249/cdqn` GitHub repository.
        *   Configure the GitHub Actions workflows (`ci-main.yml`, `docs.yml`) for automated testing, linting, formatting, and security audits in Rust.
    2.  **Kernel Primitives (`cdqnK`)**:
        *   Implement the core data structures in Rust: `KDU`, `FQEI`, `License`, etc.
        *   Implement `K.CryptoCore`: The root service for all cryptographic operations (signing, verification).
        *   Implement `K.HLC`: The Hybrid Logical Clock for generating unique, time-ordered KDU IDs.
        *   Implement `K.KDUFactory`: The authoritative service for creating and validating well-formed KDUs.
    3.  **Core Runtime (`Cmodules`)**:
        *   Implement `C.Persistence`: A basic, in-memory version for storing and retrieving KDUs by ID.
        *   Implement `C.EntityScheduler`: The core non-blocking event loop for managing entity mailboxes and executing their behaviors locally.
        *   Implement `C.ModuleLoader`: The service responsible for loading and initializing modules.
    4.  **`cdqnLang` Toolchain**:
        *   Develop the `C.Compiler` module.
        *   Implement a transpiler for a core subset of `cdqnLang` syntax: declarations, assignments, `entity` blocks, and `behavior` blocks.
    5.  **"Hello, Node" Application**:
        *   Create a simple `Umodule` in `cdqnLang` containing a `bot` entity that responds to a "ping" KDU with a "pong" KDU.
*   **Exit Criteria**: A developer can write a simple multi-entity application in `cdqnLang`, compile it using the `C.Compiler`, and run it on a single, local `cdqnRuntime` instance. All KDU-based communication between entities is local and managed by the `C.EntityScheduler`.

### **3. Phase 1: The Decentralized Network (Two-Node MVP)**

*   **Goal**: To enable two sovereign `cdqnRuntime` nodes to discover each other, establish a secure connection, and communicate reliably. This phase tackles the core challenges of decentralization.
*   **Key Milestones**:
    1.  **Networking Layer (`cdqNetwork`)**:
        *   Implement the `cdqnProt` protocol using a library like `quinn` for a QUIC-based transport layer.
        *   Implement the "Sealed Envelope" protocol for secure, private message exchange.
    2.  **Onboarding & Trust**:
        *   Implement the "Web of Trust" Onboarding Protocol, allowing a new node to be sponsored by an existing one.
        *   Implement the `OriginKDU` creation and verification ceremony.
    3.  **Distributed Runtime**:
        *   Upgrade the `C.EntityScheduler` to distinguish between local and remote FQEIs, routing KDUs to the network layer when necessary.
        *   Upgrade `C.Persistence` to use a high-performance, on-disk, log-structured storage engine (e.g., RocksDB or a custom implementation).
    4.  **Integration Testing**:
        *   Develop a comprehensive two-node test suite to verify secure onboarding, remote KDU delivery, and resilience to basic network failures.
*   **Exit Criteria**: Two `cdqnRuntime` instances can be started on separate machines. Node A can generate an invitation URI to sponsor Node B. Node B can successfully join the network. An entity on Node A can send a KDU to an entity on Node B, and vice-versa.

### **4. Phase 2: The Analytical Engine & Core Intelligence**

*   **Goal**: To build the foundational analytical capabilities of the `K.Unisphere` and the core services that provide and consume this intelligence.
*   **Key Milestones**:
    1.  **Unisphere Implementation**:
        *   Implement the `K.Unisphere` and `K.SphereFactory` modules.
        *   Implement the `Semantic Nexus` and `Ethical Compass` Core Spheres. For the MVP, this can be achieved by creating a secure, sandboxed interface to a powerful external LLM.
    2.  **Core Service Implementation**:
        *   Implement `C.Metrics` to collect and emit telemetry KDUs about node interactions.
        *   Implement `C.Trust` and the `ReputationBot`, using data from `C.Metrics` to calculate initial reputation vectors (Consistency, Merit).
        *   Implement `C.Sphere.NodeReality` to provide a basic simulation sandbox.
    3.  **Immune System (Alpha)**:
        *   Implement an initial version of `K.RedTeam` and `C.RedTeam` with a small set of predefined adversarial tests to run against other modules.
*   **Exit Criteria**: A node can send a KDU to the `K.SphereFactory` for analysis and receive a 42-dimensional vector. The `ReputationBot` can successfully update a peer's reputation score based on a series of successful and failed interactions.

---

### **5. Phase 3: The Economy, Governance, & User Experience**

*   **Goal**: To implement the economic incentives, governance loops, and user-facing components that enable a self-sustaining and usable ecosystem.
*   **Key Milestones**:
    1.  **Economic Primitives**:
        *   Implement the `C.AssetsManager` and the `cdqnStar` data structure.
        *   Implement the `Triadic Notary Protocol` for the decentralized minting of `cdqnStar`s based on a `ProofOfEventKDU`.
    2.  **Governance & Learning Loop**:
        *   Implement the `C.Licensing` module to enforce `BaDaaS` license terms.
        *   Implement the `C.Curation` module and the full workflow for submitting, analyzing, and acting upon `Proposal` and `Feedback` KDUs.
    3.  **User Interface**:
        *   Develop the `C.Interface` module (Sovereign Terminal).
        *   Develop the first version of the `ProxyAgent`, the user's primary interface to the ecosystem.
    4.  **Growth Loop**:
        *   Implement the `Public Gateway Protocol` to allow new users to onboard via `.sealed_kdu` files discovered on the public web.
*   **Exit Criteria**: Two users can successfully execute a barter, have it notarized by a third peer, and receive newly minted `cdqnStar`s. A user can use their `ProxyAgent` via the `C.Interface` to interact with the ecosystem.

---

### **6. Phase 4: Hardening, Optimization, & Adoption**

*   **Goal**: To transition the ecosystem from a functional prototype to a stable, secure, and developer-friendly platform ready for real-world use.
*   **Key Milestones**:
    1.  **Performance & Scaling**: Conduct rigorous performance profiling and optimize bottlenecks in the runtime, persistence, and network layers.
    2.  **Security Audits**: Engage external security firms to perform comprehensive audits of the entire codebase. Launch a public bug bounty program.
    3.  **Developer Experience**: Write extensive documentation, tutorials, and examples. Refine the `cdqnLang` syntax and compiler diagnostics.
    4.  **Ecosystem Growth**: Implement the remaining Core Spheres. Build out key `S` (Standard) modules. Foster a community of early adopters and contributors.
*   **Exit Criteria**: The `cdqn` ecosystem is a stable, secure, and well-documented platform with a growing community, ready for a version 1.0.0 release.
